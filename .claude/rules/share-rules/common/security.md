# セキュリティガイドライン

---

## 基本原則

### セキュリティファースト
- セキュリティは後付けで追加してはならない。設計段階から要件として取り込むこと
- 「動作する」ことよりも「安全に動作する」ことを優先すること

### 最小権限の原則
- ユーザー、サービス、プロセスには業務上必要最小限の権限のみを付与すること
- 管理者権限は必要な場合に限り、かつ必要な期間に限り使用すること
- 不要となった権限は速やかに剥奪すること

---

## 秘密情報管理

### 環境変数の使用

**秘密情報は必ず環境変数で管理しなければならない**

- APIキー、パスワード、トークン、接続文字列は環境変数で管理すること（ハードコードを禁ずる）
- `.env` ファイルは必ず `.gitignore` に追加すること
- クライアント公開が必要な場合に限り `NEXT_PUBLIC_` 等のプレフィックスを使用すること（不要な公開を禁ずる）
- 本番環境の秘密情報はシークレット管理サービス（例: AWS Secrets Manager 等）で管理すること

### 秘密情報の露出防止

- ログ、コンソール、エラーメッセージに秘密情報を出力してはならない
- スタックトレース等に秘密情報が含まれないよう、例外メッセージの組み立てを含めて留意すること
- デバッグ目的であっても秘密情報のハードコードを禁ずる

```
// Bad - 絶対禁止
const API_KEY = "sk-1234567890abcdef";
console.log(`Token: ${authToken}`);

// Good
const API_KEY = process.env.API_KEY;
console.log("Token received"); // 値は出力しない
```

### モノレポでの秘密情報分離

- 各サービスは原則として独自の `.env` を保有し、スコープ外の秘密情報を参照してはならない
- 共通の秘密情報はルートレベルで管理し、必要なサービスのみに明示的に供給すること
- サービス間で秘密情報を共有する場合は、共有範囲・用途・参照方法を明示的に文書化すること

## 依存関係のセキュリティ

### 脆弱性管理

- 依存パッケージは定期的に更新し、既知の脆弱性を放置してはならない
- `npm audit` / `pnpm audit` を定期的に実行すること
- 重大な脆弱性（Critical/High）は最優先で対応すること
- 修正が困難な場合は、代替パッケージへの移行または当該機能の利用停止を検討すること

```bash
# 脆弱性チェック
pnpm audit

# 自動修正可能な脆弱性を修正
pnpm audit --fix
```

### 依存関係の追加時

- 新規パッケージ追加前に、セキュリティ面の妥当性を評価すること
- メンテナンスが停止しているパッケージの採用を原則として禁ずる
- ダウンロード数が極端に少ない、または信頼性に乏しいパッケージは慎重に検討すること
- ライセンスを確認し、利用条件に抵触しないことを保証すること

### ロックファイルの管理

- `pnpm-lock.yaml` / `package-lock.json` は必ずコミットすること
- ロックファイルの差分は意図した変更であることを確認すること
- CI/CDではロックファイルに従ってインストールすること（例: `pnpm install --frozen-lockfile`）

## 入力検証

### 基本ルール

**すべての外部入力は信頼してはならない**

- ユーザー入力、APIリクエスト、外部サービス由来データは必ず検証すること
- バリデーションはサーバーサイドで必ず実施すること（クライアントサイドは補助に過ぎない）
- 型チェックに加え、値の範囲・形式・長さ等も検証対象とすること

### 検証の実装

```typescript
// Good - 入力を検証してから使用
const userId = validateUserId(request.params.id);
if (!userId) {
  throw new BadRequestError("Invalid user ID");
}

// Bad - 入力をそのまま使用
const userId = request.params.id; // 検証なし
```

### SQLインジェクション対策

- プリペアドステートメント／パラメータ化クエリを必ず使用すること
- ORMを使用していても raw クエリの利用時は特に注意すること
- ユーザー入力をSQL文字列へ直接連結してはならない

### XSS対策

- ユーザー入力をHTMLへ出力する際は必ずエスケープすること
- フレームワークの自動エスケープ機構を適切に利用すること
- `dangerouslySetInnerHTML` 等は必要最小限に留め、使用時は必ずサニタイズすること

## 認証・認可

### 認証情報の取り扱い

- Cookie-based 認証では `credentials: 'include'` を必須とすること
- CSRF対策として、POST/PUT/PATCH/DELETE は `X-CSRF-Token` ヘッダーを必須とすること
- セッショントークンは HttpOnly / Secure / SameSite 属性を適切に設定すること
- パスワードは bcrypt 等で適切にハッシュ化し、平文保存を禁ずる

### 認可チェック

- すべてのエンドポイントで認可チェックを実施しなければならない
- ユーザーID改ざん等により他ユーザーのデータへアクセスできないことを保証すること
- 管理者機能へのアクセスは厳格に制御し、誤付与・過剰権限を防止すること

```typescript
// Good - 認可チェックを実施
if (resource.ownerId !== currentUser.id && !currentUser.isAdmin) {
  throw new ForbiddenError("Access denied");
}

// Bad - 認可チェックなし
return resource; // 誰でもアクセス可能
```

## 通信のセキュリティ

### HTTPS必須

- 本番環境ではHTTPSを必須とすること
- 開発環境であっても可能な限りHTTPSを使用すること
- HTTP経由で秘密情報を送信してはならない

### CORS設定

- 許可するオリジンは明示的に指定すること（`*` を禁ずる）
- credentials を使用する場合は設定を特に厳密に行うこと
- プリフライトリクエストを適切に処理すること

## ログとモニタリング

### セキュリティログ

- 認証の成功・失敗を記録すること
- 権限エラーおよび不正アクセス試行を記録すること
- ログへ秘密情報を含めてはならない

### モニタリング

- 異常なアクセスパターンを検知できる仕組みを導入すること
- レート制限を実装し、ブルートフォース攻撃を防止すること
- 重大なセキュリティイベントに対してアラートを設定すること

---

## CI/CDパイプラインのセキュリティ

### シークレット管理

- CI/CD環境の秘密情報はシークレット変数として管理すること
- ログへシークレットが出力されないよう設定すること
- シークレットへのアクセス権限は最小化すること

### パイプラインの保護

- ブランチ保護ルールを設定すること
- プルリクエストのレビューを必須化すること
- 信頼できないコードが自動実行されないよう制御すること

---

## 禁止事項（絶対禁止）

### 秘密情報関連
- APIキー・パスワード・トークンのハードコード
- 秘密情報のログ出力
- 秘密情報を含むファイルのコミット
- 本番環境の秘密情報を開発環境で使用すること

### コード関連
- 入力検証なしで外部データを使用すること
- SQL文字列を直接連結すること
- 未サニタイズのユーザー入力をHTMLへ出力すること
- 認可チェックなしでリソースにアクセスさせること

### 設定関連
- 本番環境における CORS `origin: '*'` の使用
- 本番環境でデバッグモードを有効化すること
- 脆弱性のある依存パッケージを放置すること

**上記の違反は重大なセキュリティインシデントに直結し得るため、厳守すること**

---

## インシデント対応

### 発見時の対応

1. **即時報告**: セキュリティ上の問題（疑いを含む）を発見した場合は直ちに報告すること
2. **影響範囲の特定**: 影響を受け得るデータ・システム・ユーザー範囲を特定すること
3. **封じ込め**: 被害拡大防止のための緊急措置を実施すること
4. **記録**: 発見から対応までの経緯を記録し、再発防止に活用すること

### 報告すべき事象

- 秘密情報の漏洩（疑いを含む）
- 不正アクセスの痕跡
- 脆弱性の発見
- 異常なシステム挙動

**セキュリティインシデントの隠蔽を禁ずる**
