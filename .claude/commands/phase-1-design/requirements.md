$ARGUMENTS

---

上記のサービス概要から、段階的に要件定義書を作成してください。

## あなたの役割
経験豊富なシステムアーキテクト兼プロダクトマネージャー

## 実行方法
- このタスクは **ultrathink** で実行すること
- **Phase 1-A -> 承認 -> Phase 1-B** の順で段階的に進めること
- 各Phaseの出力後、ユーザーの承認を待ってから次に進むこと

---

## Phase 1-A: ビジネス要件（承認必須）

### 出力ファイル
`docs/requirements/business-requirements.md`

### 出力内容

#### 1. プロジェクト概要
- プロジェクト名
- 一言サマリー（エレベーターピッチ形式）
- ターゲットユーザー
- 解決する課題

#### 2. 背景・目的（Why）
- 背景: なぜこのサービスが必要か
- 現状の課題: 既存の解決策の問題点
- 目的: 何を実現したいか
- 期待効果: 定量的・定性的な効果

#### 3. 機能要件サマリー（What）

##### 3.1 ユーザーストーリー一覧
| ID | ユーザー種別 | やりたいこと | 理由 | 優先度 |
|----|------------|------------|------|-------|
| US-001 | ... | ... | ... | Must/Should/Could |

##### 3.2 機能一覧（概要）
| ID | 機能名 | 説明 | 関連US | 優先度 |
|----|-------|-----|-------|-------|
| F-001 | ... | ... | US-XXX | P0/P1/P2 |

#### 4. ビジネスリスク・課題
| ID | リスク/課題 | 影響度 | 発生確率 | 対策 |
|----|-----------|-------|---------|-----|
| BR-001 | ... | 高/中/低 | 高/中/低 | ... |

### Phase 1-A 完了後のアクション
```
Phase 1-A（ビジネス要件）を出力しました。
内容を確認し、問題なければ「承認」と入力してください。
修正がある場合は修正内容を指示してください。
```

---

## Phase 1-B: 機能詳細仕様（Phase 1-A 承認後に実行）

### 出力ファイル
**機能ごとにディレクトリを作成し、関連ドキュメントをまとめる**

#### ディレクトリ構成
```
docs/requirements/
├── business-requirements.md
└── functions/
    ├── _index.md                      # 機能一覧・依存関係・MVP定義
    │
    ├── auth/                          # 認証関連
    │   ├── F-001-login.md
    │   └── F-002-logout.md
    │
    ├── [category]/                    # カテゴリ別
    │   └── F-XXX-[function].md
    └── ...
```

#### 命名規則
- カテゴリディレクトリ: `[カテゴリ名のkebab-case]/`
- 機能ファイル: `F-XXX-[機能名のkebab-case].md`
- 機能IDはプロジェクト全体で一意の連番とする

### 出力内容

#### 1. インデックスファイル（_index.md）

```markdown
# 機能詳細仕様 インデックス

> 関連: [ビジネス要件](../business-requirements.md)

## カテゴリ別機能一覧

### [カテゴリ名]
| ID | 機能名 | 仕様 | 優先度 | ステータス |
|----|-------|-----|-------|----------|
| F-001 | ... | [F-001-xxx.md](./[category]/F-001-xxx.md) | P0 | Draft |

## 機能間の依存関係
[依存関係図を記述]

## MVP機能セット定義
| フェーズ | 含まれる機能 | 目的 |
|---------|------------|-----|
| MVP | F-001, F-002, ... | 最小限の価値提供 |
| v1.1 | ... | 機能拡張 |
```

#### 2. 機能詳細ファイル（F-XXX-[機能名].md）

**P0機能すべて** について以下のテンプレートで作成:

```markdown
# F-XXX: [機能名]

> 関連: [ビジネス要件](../../business-requirements.md) | [機能一覧](../_index.md)
> カテゴリ: [カテゴリ名]

## 基本情報

| 項目 | 内容 |
|-----|-----|
| 機能ID | F-XXX |
| 優先度 | P0 |
| 関連US | US-XXX, US-YYY |
| ステータス | Draft |

## 概要
この機能が解決する具体的な課題を記述

## ユーザーフロー
1. **[トリガー]** ユーザーが〇〇する
2. **[システム]** △△を表示/処理する
3. **[ユーザー]** □□を入力/選択する
4. **[システム]** 結果を◇◇する
5. **[完了]** ××が達成される

## 入力/出力
| 種別 | 項目名 | 型 | 必須 | 説明 | 制約・バリデーション |
|-----|-------|---|-----|-----|-------------------|
| 入力 | ... | string | Yes | ... | 最大XX文字 |
| 出力 | ... | ... | ... | ... | ... |

## ビジネスルール
| ID | ルール | 条件 | 結果 |
|----|-------|-----|-----|
| BR-FXXX-01 | ... | ...の場合 | ...する |

## エラーケース
| ID | 条件 | エラーメッセージ | 対処 |
|----|-----|----------------|-----|
| E-FXXX-01 | ... | ... | ... |

## 受け入れ条件（AC）
- [ ] AC-01: 〇〇した場合、△△が表示されること
- [ ] AC-02: □□を入力した場合、◇◇が保存されること
- [ ] AC-03: 異常系：××の場合、エラーメッセージが表示されること

## 未確定事項
- 【要確認】〇〇の仕様は要確認
- 【仮定】△△は□□と仮定
```

### 実行方法
- **subAgent** で各カテゴリの機能ファイルを並列出力すること
- _index.md は全機能ファイル出力後に作成

### Phase 1-B 完了後のアクション
```
Phase 1-B（機能詳細仕様）を出力しました。

出力ディレクトリ構成:
docs/requirements/
├── business-requirements.md
└── functions/
    ├── _index.md
    └── [カテゴリ]/
        └── F-XXX-[機能名].md

内容を確認し、問題なければ次のステップ（phase-1-design/architecture）を実行してください。
修正がある場合は、対象の機能IDと修正内容を指示してください。
```

---

## 出力ルール
- 不明点は「【要確認】」タグを付ける
- 仮定を置いた場合は「【仮定】」タグを付ける
- 優先度は Must/Should/Could または P0/P1/P2 で統一
- 機能IDはプロジェクト全体で一意の連番（F-001, F-002, ...）
- カテゴリディレクトリ名・機能ファイル名は kebab-case で統一
