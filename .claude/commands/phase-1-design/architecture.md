# Phase 1-B: アーキテクチャ設計

## 前提
以下のファイルが完了済みであること:
- `docs/requirements/business-requirements.md`
- `docs/requirements/functions/_index.md`

---

## 実行指示

以下のファイルを読み込み、アーキテクチャ設計と非機能要件を作成してください:
- `docs/requirements/business-requirements.md`
- `docs/requirements/functions/_index.md`

詳細仕様が必要な場合は `docs/requirements/functions/` 配下の各ファイルを参照

## あなたの役割
経験豊富なシステムアーキテクト。
最新の技術トレンドを把握し、プロジェクトに最適な構成を提案できる。

## 実行方法
このタスクは **ultrathink** で実行すること。

---

## Step 1: 技術スタック提案

business-requirements.md と functions/_index.md を読み込んだ後、**2パターンの技術構成を提案**する。

### 提案フォーマット

```
要件を確認しました。以下の2パターンで技術構成を提案します。

---

## パターンA: MVP優先（シンプル・低コスト・高速開発）

**コンセプト**: 最小限の技術スタックで素早くリリース。学習コスト低め。

| カテゴリ | 技術 | 理由 |
|---------|------|------|
| フロントエンド | Next.js (App Router) | Reactエコシステム、SSR/SSG対応 |
| スタイリング | Tailwind CSS | 高速開発、カスタマイズ容易 |
| UIコンポーネント | shadcn/ui | コピペ可能、依存少ない |
| バックエンド | Next.js API Routes | フロントと統合、デプロイ簡単 |
| 認証 | Supabase Auth | 無料枠あり、設定簡単 |
| データベース | Supabase (PostgreSQL) | 認証と統合、無料枠あり |
| ホスティング | Vercel | Next.js最適化、無料枠あり |

**メリット**: 開発速度◎、コスト◎、学習コスト◎
**デメリット**: 大規模化時にリファクタリング必要

---

## パターンB: リッチ構成（拡張性・保守性重視）

**コンセプト**: 中長期運用を見据えた堅牢な構成。チーム開発向き。

| カテゴリ | 技術 | 理由 |
|---------|------|------|
| フロントエンド | Next.js (App Router) | Reactエコシステム、SSR/SSG対応 |
| 状態管理 | Zustand + TanStack Query | 軽量、キャッシュ管理 |
| バックエンド | Hono (on Cloudflare Workers) | 高速、エッジ対応 |
| API形式 | tRPC | 型安全、DX向上 |
| 認証 | Auth.js (NextAuth) | 柔軟性高、プロバイダー豊富 |
| データベース | Neon (PostgreSQL) | サーバーレス、ブランチ機能 |
| ORM | Drizzle | 軽量、型安全 |

**メリット**: 拡張性◎、型安全◎、パフォーマンス◎
**デメリット**: 学習コストやや高め、初期構築に時間

---

## どちらを選びますか？

1. **A（MVP優先）で進める**
2. **B（リッチ構成）で進める**
3. **ベースを選んで一部変更**（例: 「Aベースで認証だけFirebaseに」）
4. **独自の技術スタックを指定**（使いたい技術があれば教えてください）
```

### 提案時の注意

- 上記はテンプレート例。business-requirements.md の内容に応じて技術を調整すること
- 要件に特殊な要素（リアルタイム通信、大量ファイル処理など）があれば、それに対応する技術を含める
- 各技術の選定理由は要件と紐づけて説明する

---

## Step 2: 調査・確定

ユーザーの選択・指定を受けて、以下を実行:

### 2.1 技術スタックの確定

| ユーザーの選択 | 対応 |
|--------------|------|
| パターンA or Bを選択 | 選択されたパターンをベースに進める |
| 一部変更の指示 | 指定された技術に差し替え |
| 独自技術の指定 | 指定された技術で構成を組み直す |

### 2.2 Web検索で最新情報を確認

- 確定した技術の現在のバージョン・状況
- 技術の組み合わせにおけるベストプラクティス
- 類似プロジェクトでの採用事例

### 2.3 ultrathinkで構成を検討

- business-requirements.md の要件との適合性
- ユーザーの選択との整合性
- トレードオフの評価
- 不足している技術の補完（テスト、監視など）

---

## Step 3: 全体構成の提案

調査結果を踏まえ、**2つのファイルを一括で提案**する。
