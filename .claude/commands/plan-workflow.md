# Plan Workflow Runner

計画の作成から実装完了までを段階的にガイドし、適切なタイミングで `/clear` を促すオーケストレーター。

## 使用方法

```bash
# 新規計画から開始
/plan-workflow "機能の説明"

# 既存計画の進捗確認・続行
/plan-workflow <plan-name>

# 途中から再開
/plan-workflow <plan-name> --from impl

# 進捗確認のみ
/plan-workflow <plan-name> --status
```

## 引数

$ARGUMENTS

- `<description>`: 新規計画の場合、実装したい内容の説明
- `<plan-name>`: 既存計画の場合、計画ファイル名
- `--from <step>`: 指定ステップから再開 (plan|impl|verify)
- `--status`: 進捗確認のみ（実行なし）

## ワークフロー概要

```
┌─────────────────────────────────────────────────────────────┐
│                    Plan Workflow                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: PLAN (計画作成)                                    │
│    └─ /plan "<説明>" でコードベース探索・計画作成           │
│                    ↓                                        │
│              [/clear 推奨]                                  │
│                    ↓                                        │
│  Step 2: IMPL (実装)                                        │
│    └─ /run-plan <name> --chunked でフェーズ単位実行         │
│    └─ 各フェーズ完了時に /clear → 次フェーズ再開            │
│                    ↓                                        │
│              [/clear 推奨]                                  │
│                    ↓                                        │
│  Step 3: VERIFY (検証)                                      │
│    └─ テスト・ビルド・動作確認                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 実行手順

### Step 1: 進捗確認

1. 引数が説明文（新規）か計画名（既存）かを判定
2. 既存の場合、以下を確認:
   - `.claude/plans/{plan-name}.md` の存在
   - `.claude/state/{plan-name}-progress.json` の存在
   - 計画内の Tasks 完了状況

### Step 2: 状態判定

```
状態判定フロー:

1. 計画ファイルなし
   → Step: INIT (新規作成が必要)

2. 計画ファイルあり & タスク未着手
   → Step: PLAN_DONE (実装開始可能)

3. 計画ファイルあり & タスク一部完了
   → Step: IMPL_IN_PROGRESS (実装継続)

4. 計画ファイルあり & 全タスク完了 & 検証未完了
   → Step: IMPL_DONE (検証が必要)

5. 計画ファイルあり & 全タスク完了 & 検証完了
   → Step: COMPLETED (完了)
```

## コンテキスト管理戦略

### 各ステップ間で /clear する理由

| ステップ | コンテキスト消費 | /clear 推奨 |
|----------|------------------|-------------|
| PLAN | 高（コードベース探索） | 完了後に必須 |
| IMPL (各フェーズ) | 中〜高 | フェーズ完了ごとに推奨 |
| VERIFY | 低 | 不要 |

### 自己完結性の確保

計画ファイルに以下を必ず含めることで、/clear 後も作業継続可能:

1. **Goal**: なぜこの作業が必要か
2. **Files テーブル**: 対象ファイル一覧
3. **Tasks 詳細**: 具体的な実装指示
4. **進捗状態**: `.claude/state/` で永続化

## エラーハンドリング

### 計画が見つからない

```
❌ 計画が見つかりません: {plan-name}

利用可能な計画:
  - 2024-01-15-login-validation (IMPL: 3/5 完了)
  - 2024-01-16-dashboard-refactor (PLAN_DONE)
  ...

新規作成する場合:
  /plan-workflow "実装したい内容の説明"
```

### 実装中にエラー

```
⚠️ 実装でエラーが発生しました

最後に成功したタスク: 2.2
エラーが発生したタスク: 2.3

進捗は保存されています。修正後に再開:
  /run-plan {plan-name} --from 2.3
```

## 完了オプション

### --complete フラグ

検証完了を記録し、計画を完了状態にする:

```bash
/plan-workflow {plan-name} --complete
```

実行内容:
1. 計画ファイルのステータスを `completed` に更新
2. `Completed: {timestamp}` を追加
3. 進捗ファイルに完了を記録

## 補足

### セッション間の継続

ワークフローの状態は以下で管理:
- **計画ファイル**: `.claude/plans/{name}.md`
- **進捗状態**: `.claude/state/{name}-progress.json`

いつでもセッションを閉じて、後から再開可能:

```bash
# 新セッションで進捗確認
/plan-workflow my-feature --status

# 続きから実行
/plan-workflow my-feature
```

### 推奨ワークフロー例

```bash
# Session 1: 計画作成
/plan-workflow "ユーザープロフィール編集機能を追加"
/plan "ユーザープロフィール編集機能を追加"
# → .claude/plans/2024-01-17-user-profile-edit.md 作成

# Session 2: 実装 Phase 1-2
/clear
/plan-workflow 2024-01-17-user-profile-edit
/run-plan 2024-01-17-user-profile-edit --chunked
# → Phase 1, 2 完了

# Session 3: 実装 Phase 3-4
/clear
/plan-workflow 2024-01-17-user-profile-edit
/run-plan 2024-01-17-user-profile-edit --from 3.1
# → Phase 3, 4 完了

# Session 4: 検証・完了
/clear
/plan-workflow 2024-01-17-user-profile-edit
# → 検証実行
/plan-workflow 2024-01-17-user-profile-edit --complete
```
