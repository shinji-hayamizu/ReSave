---

上記のサービス概要から、段階的に要件定義書を作成してください。

## あなたの役割
経験豊富なシステムアーキテクト兼プロダクトマネージャー

## 実行方法
- このタスクは **ultrathink** で実行すること
- **Phase 1-A → 承認 → Phase 1-B** の順で段階的に進めること
- 各Phaseの出力後、ユーザーの承認を待ってから次に進むこと

---

## Phase 1-A: ビジネス要件（承認必須）

### 出力ファイル
`docs/requirements/business-requirements.md`

### 出力内容

#### 1. プロジェクト概要
- プロジェクト名
- 一言サマリー（エレベーターピッチ形式）
- ターゲットユーザー
- 解決する課題

#### 2. 背景・目的（Why）
- 背景: なぜこのサービスが必要か
- 現状の課題: 既存の解決策の問題点
- 目的: 何を実現したいか
- 期待効果: 定量的・定性的な効果

#### 3. 機能要件サマリー（What）

##### 3.1 ユーザーストーリー一覧
| ID | ユーザー種別 | やりたいこと | 理由 | 優先度 |
|----|------------|------------|------|-------|
| US-001 | ... | ... | ... | Must/Should/Could |

##### 3.2 機能一覧（概要）
| ID | 機能名 | 説明 | 関連US | 優先度 |
|----|-------|-----|-------|-------|
| F-001 | ... | ... | US-XXX | P0/P1/P2 |

#### 4. ビジネスリスク・課題
| ID | リスク/課題 | 影響度 | 発生確率 | 対策 |
|----|-----------|-------|---------|-----|
| BR-001 | ... | 高/中/低 | 高/中/低 | ... |

### Phase 1-A 完了後のアクション
```
Phase 1-A（ビジネス要件）を出力しました。
内容を確認し、問題なければ「承認」と入力してください。
修正がある場合は修正内容を指示してください。
```

---

## Phase 1-B: 機能詳細仕様（Phase 1-A 承認後に実行）

### 出力ファイル
**機能ごとに個別ファイル** を作成する

#### ディレクトリ構成
```
docs/requirements/
├── business-requirements.md
└── functions/
    ├── _index.md          # 機能一覧・依存関係・MVP定義
    ├── F-001-[機能名].md
    ├── F-002-[機能名].md
    └── ...
```

#### ファイル命名規則
`F-XXX-[機能名のkebab-case].md`
例: `F-001-user-auth.md`, `F-002-dashboard.md`

### 出力内容

#### 1. インデックスファイル（_index.md）

```markdown
# 機能詳細仕様 インデックス

> 関連: [ビジネス要件](../business-requirements.md)

## 機能一覧

| ID | 機能名 | ファイル | 優先度 | ステータス |
|----|-------|---------|-------|----------|
| F-001 | ユーザー認証 | [F-001-user-auth.md](./F-001-user-auth.md) | P0 | Draft |
| F-002 | ダッシュボード | [F-002-dashboard.md](./F-002-dashboard.md) | P0 | Draft |

## 機能間の依存関係

F-001 (認証)
  └→ F-002 (ダッシュボード) 
      ├→ F-003 (データ登録)
      └→ F-004 (レポート出力)

## MVP機能セット定義

| フェーズ | 含まれる機能 | 目的 |
|---------|------------|-----|
| MVP | F-001, F-002, F-003 | 最小限の価値提供 |
| v1.1 | F-004, F-005 | 〇〇の拡張 |
| v1.2 | F-006 | △△の対応 |
```

### 実行方法
- **subAgent** で各機能ファイルを並列出力すること
- _index.md は全機能ファイル出力後に作成

### Phase 1-B 完了後のアクション
```
Phase 1-B（機能詳細仕様）を出力しました。

出力ファイル:
- docs/requirements/functions/_index.md
- docs/requirements/functions/F-001-xxx.md
- docs/requirements/functions/F-002-xxx.md
- ...

内容を確認し、問題なければ次のステップ（02-create-architecture）を実行してください。
修正がある場合は、対象の機能IDと修正内容を指示してください。
```

## 出力ルール
- 不明点は「【要確認】」タグを付ける
- 仮定を置いた場合は「【仮定】」タグを付ける
- 優先度は Must/Should/Could または P0/P1/P2 で統一
- 各機能ファイルは business-requirements.md の機能IDと整合させる
- ファイル名は kebab-case で統一（例: user-auth, data-export）
