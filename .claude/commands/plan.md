# Plan Creator

タスクの実装計画を作成し、別セッションで実行可能な形式で保存します。

## 使用方法

```
/plan <タスク説明>
/plan <タスク説明> --name <plan-name>
```

## 引数

$ARGUMENTS

- `<タスク説明>`: 実装したい内容の説明（必須）
- `--name <plan-name>`: 計画ファイル名（省略時は自動生成）

## 実行手順

### Step 1: 引数解析

引数から以下を抽出:
- タスク説明: 最初の引数
- `--name`: 計画ファイル名（指定がない場合は説明から slug を生成）

### Step 2: コードベース探索

**計画作成に必要な情報を収集**:

1. **関連ファイルの特定**
   - タスク説明からキーワードを抽出
   - Glob/Grep で関連ファイルを検索
   - 既存の実装パターンを把握

2. **依存関係の確認**
   - 変更対象ファイルの import/export を確認
   - 影響を受ける可能性のあるファイルを特定

3. **既存パターンの参照**
   - 類似機能の実装を検索
   - プロジェクトのコーディング規約を確認

### Step 3: 計画ドキュメント作成

以下のフォーマットで `.claude/plans/YYYY-MM-DD-{name}.md` を作成:

```markdown
# Plan: {タイトル}

> Created: {YYYY-MM-DD HH:mm}
> Status: draft

## Summary

{1-2行の概要}

## Goal

{達成したいこと - Why}

## Context

### Background
{タスクの背景・動機}

### Constraints
{制約・注意点}

### References
{参照すべきドキュメント・ファイル}

## Analysis

### Current State
{現在の実装状況}

### Target State
{実装後の状態}

### Impact
{変更の影響範囲}

## Files

| Path | Role | Action |
|------|------|--------|
| {パス} | {役割} | {create/modify/delete/reference} |

## Tasks

### Phase 1: {フェーズ名}

- [ ] **1.1** {タスク名}
  - Target: `{対象ファイル}`
  - Action: {具体的な作業内容}
  - Details:
    - {実装詳細1}
    - {実装詳細2}
  - Acceptance: {完了条件}

- [ ] **1.2** {タスク名}
  ...

### Phase 2: {フェーズ名}
...

## Verification

- [ ] 単体テスト: `{テストコマンド}`
- [ ] 型チェック: `pnpm type-check` (該当する場合)
- [ ] ビルド確認: `pnpm build`
- [ ] 動作確認: {確認手順}

## Notes

{実装時の注意点・代替案・懸念事項}
```

### Step 4: 自己完結性の検証

計画が別セッションで実行可能か検証:

1. **コンテキスト完全性チェック**
   - 対象ファイルパスが明記されているか
   - 実装詳細が曖昧でないか
   - 参照パターンが特定されているか

2. **タスク粒度チェック**
   - 各タスクが1コンテキストで完結するサイズか
   - 依存関係が明確か

3. **不足情報の補完**
   - 曖昧な箇所があれば具体化
   - 必要なコードスニペットを追加

### Step 5: 出力

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Plan created: .claude/plans/{filename}.md

Summary:
  {概要}

Structure:
  - Phases: {N}
  - Tasks: {M}
  - Files: {K}

Next Steps:
  1. Review the plan: Read .claude/plans/{filename}.md
  2. Approve and execute: /run-plan {filename}

Or in a new session:
  /run-plan {filename}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 重要な原則

### コンテキスト自己完結性

**別セッションで実行できるよう、以下を必ず含める:**

1. **Why（理由）**: なぜこの変更が必要か
2. **What（対象）**: 具体的なファイルパスと変更内容
3. **How（方法）**: 実装の具体的手順
4. **Reference（参照）**: 既存パターンの場所

### タスク粒度

- 1タスク = 1-3時間程度の作業量
- 1タスク = 1セッションで完結
- 依存タスクは明示的に記載

### 曖昧さの排除

以下は禁止:
- 「適切に実装する」
- 「必要に応じて修正」
- 「同様の処理を追加」

代わりに具体的に記述:
- 「loginSchema に email と password フィールドを追加」
- 「useForm に zodResolver(loginSchema) を設定」

## エラーハンドリング

### タスク説明が不明確な場合

```
⚠️ タスク説明が不明確です

以下の情報を追加してください:
- 具体的な機能名または画面名
- 期待する動作
- (任意) 参考となる既存機能

例: /plan "ログイン画面にメールアドレスのバリデーションを追加"
```

### 関連ファイルが見つからない場合

```
⚠️ 関連ファイルの特定に失敗

検索キーワード: {keywords}
検索結果: 0件

以下を試してください:
1. より具体的なファイル名やパスを指定
2. 関連する機能名を追加
```

## 補足

### 計画ファイルの管理

- 保存先: `.claude/plans/`
- 命名規則: `YYYY-MM-DD-{slug}.md`
- ステータス: draft → approved → completed

### 既存計画との統合

同名の計画が存在する場合:
1. 既存計画を読み込み
2. マージまたは上書きを確認
3. バージョン管理（必要に応じて）

## ワークフロー連携

### 推奨フロー

計画作成後は `/plan-workflow` でワークフロー全体を管理:

```bash
# 1. 計画作成
/plan "ユーザープロフィール編集機能"

# 2. ワークフロー開始（進捗管理・実装・検証）
/plan-workflow 2024-01-17-user-profile-edit
```

### 次のステップ案内

計画作成完了時の出力に以下を追加:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Plan created: .claude/plans/{filename}.md

...

Workflow Options:
  A. 今すぐ実装開始:
     /run-plan {filename} --chunked

  B. ワークフローで管理（推奨）:
     /clear
     /plan-workflow {filename}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
