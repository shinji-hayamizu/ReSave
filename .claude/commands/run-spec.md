# 汎用 Spec タスクランナー

指定された spec のタスクを自動実行します。

## 使用方法

```
/run-spec <spec-name>
/run-spec <spec-name> --from <task-id>
/run-spec <spec-name> --only <task-ids>
/run-spec <spec-name> --dry-run
```

## 引数

$ARGUMENTS

## 実行手順

### Step 1: 引数解析

引数から以下を抽出:
- `spec-name`: 必須。spec ディレクトリ名
- `--from <id>`: 指定タスクから再開
- `--only <ids>`: カンマ区切りで指定タスクのみ実行
- `--dry-run`: 実行計画のみ表示
- `--parallel`: 並列実行可能タスクを同時実行（デフォルト: true）

### Step 2: Spec ファイル読み込み

以下のパスを順に検索:
1. `.kiro/specs/{spec-name}/tasks.md`
2. `.claude/specs/{spec-name}/tasks.md`
3. `docs/specs/{spec-name}/tasks.md`

### Step 3: タスク構造解析

tasks.md から以下を抽出:
- `- [ ] X.Y タスク名` 形式のタスク一覧
- `(P)` マーク付きタスクは並列実行可能
- インデント配下の箇条書きは実行詳細
- `_Requirements: X.X, ..._` は関連要件

タスクをフェーズでグループ化:
- 同一メジャー番号（X.Y の X）は同一フェーズ
- フェーズ内の (P) タスクは並列実行候補

### Step 4: 実行計画生成

```
📋 実行計画: {spec-name}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Phase 1: {フェーズ名}
  ├─ 1.1 {タスク名}

Phase 2: {フェーズ名}
  ├─ 2.1 {タスク名}
  └─ 2.2 {タスク名}

Phase 3: {フェーズ名} [並列実行]
  ├─ 3.1 (P) {タスク名} ─┐
  ├─ 3.2 (P) {タスク名} ─┼─ 同時実行
  └─ 3.3 (P) {タスク名} ─┘

総タスク数: X | 推定フェーズ: Y
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

`--dry-run` 時はここで終了。

### Step 5: フェーズ順次実行

各フェーズを順次実行:

```
🚀 Phase {N} 開始: {フェーズ名}
─────────────────────────────────
```

#### 5a. 通常タスク（順次）

```
▶️ Task {X.Y}: {タスク名}
  - {詳細1}
  - {詳細2}
  ...
✅ Task {X.Y} 完了
```

#### 5b. 並列タスク

(P) マーク付きタスクが複数ある場合:

```
⚡ 並列実行開始 ({N}タスク)
  ├─ {X.1} {タスク名}
  ├─ {X.2} {タスク名}
  └─ {X.3} {タスク名}
```

各タスクを並列で実行し、全完了を待機。

### Step 6: 進捗記録

完了タスクを状態ファイルに記録:

`.claude/state/{spec-name}-progress.json`:
```json
{
  "spec": "{spec-name}",
  "started_at": "ISO8601",
  "updated_at": "ISO8601", 
  "tasks": {
    "1.1": { "status": "completed", "completed_at": "..." },
    "2.1": { "status": "completed", "completed_at": "..." },
    "2.2": { "status": "in_progress" }
  }
}
```

### Step 7: エラーハンドリング

タスク失敗時:
1. エラー内容を表示
2. 進捗を保存
3. 再開コマンドを提示:
   ```
   ❌ Task {X.Y} 失敗
   
   再開するには:
   /run-spec {spec-name} --from {X.Y}
   ```

### Step 8: 完了サマリー

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎉 Spec 実行完了: {spec-name}

実行結果:
  ✅ 成功: {N} タスク
  ⏭️ スキップ: {M} タスク
  ❌ 失敗: {K} タスク

生成ファイル:
  - path/to/file1.ts
  - path/to/file2.ts
  ...

所要時間: {HH:MM:SS}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 補足

### 関連ファイル自動読み込み

各 spec で以下のファイルも参照:
- `requirements.md` - 要件定義
- `design.md` - 設計仕様（存在する場合）
- プロジェクトの `CLAUDE.md` - コード規約

### コンテキスト節約

各タスク実行時、必要最小限のコンテキストのみ渡す:
- 当該タスクの詳細
- 関連する requirements
- 直前タスクの成果物パス

### 再開機能

`--from` 指定時は進捗ファイルを確認し、指定タスク以前を「完了」としてスキップ。
