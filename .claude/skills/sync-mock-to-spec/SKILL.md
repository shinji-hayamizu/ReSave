# HTMLモック → 機能仕様書 同期スキル

## 概要

HTMLモック（`docs/screens/mock/`）を分析し、機能仕様書を更新する。
**ファイル更新は必ずサブエージェントに委譲する。**

## 実行手順

### Phase 1: 分析（親エージェントで実行）

1. **モックディレクトリを走査**
   ```
   docs/screens/mock/{version}/*.html
   ```

2. **各HTMLから情報を抽出**
   - HTMLコメントから機能ID（F-XXX）を取得
   - data-component, data-action, data-state等を解析
   - 更新対象の仕様書パスを特定

3. **更新内容を整理**
   - 仕様書ごとに更新すべき内容をまとめる

### Phase 2: 更新（サブエージェントに委譲）

**1ファイルにつき1つのTaskを起動する。**

各Taskには以下を渡す:
- 対象仕様書のパス
- 抽出したUI情報（コンポーネント、アクション、状態）
- 更新すべきセクション

#### サブエージェント呼び出し例

```
Task({
  subagent_type: "general-purpose",
  description: "Update F-013 spec",
  prompt: `
仕様書を更新せよ。

対象ファイル: docs/requirements/functions/card/F-013-card-create.md

更新内容:
- 画面要件セクションに以下を追記:

### コンポーネント構成
| コンポーネント | 役割 |
|--------------|------|
| CardCreateForm | カード作成フォーム |
| FrontInput | 表面入力欄 |
| BackInput | 裏面入力欄 |

### ボタン/アクション
| 要素 | アクション |
|-----|----------|
| 保存ボタン | onClick:handleSave |

注意:
- 既存の「ビジネスルール」「技術仕様」は変更しない
- 変更履歴セクションに更新日を追記
`
})
```

### Phase 3: 結果報告

サブエージェントの完了を待ち、更新結果をまとめて報告。

## 抽出対象のdata属性

| 属性 | 用途 | 仕様書への反映 |
|------|------|---------------|
| data-component | コンポーネント名 | コンポーネント構成表 |
| data-action | ユーザーアクション | ボタン/アクション表 |
| data-prop | 状態管理プロパティ | 入力フィールド表 |
| data-show-if | 条件付き表示 | 状態バリエーション表 |
| data-state | 状態名 | 状態バリエーション表 |

## 仕様書更新フォーマット

サブエージェントに渡す更新内容は以下の形式で整理:

```markdown
### コンポーネント構成
| コンポーネント | 役割 | 子コンポーネント |
|--------------|------|----------------|

### 入力フィールド
| フィールド | 種類 | 必須 | 制約 |
|-----------|------|-----|------|

### ボタン/アクション
| 要素 | アクション | 説明 |
|-----|----------|------|

### 状態バリエーション
| 状態名 | 条件 | 表示内容 |
|-------|------|---------|
```

## 注意事項

1. **親エージェントはファイル編集しない** - Read/Glob/Grepのみ使用
2. **並列実行可能** - 依存関係のない複数Taskは同時起動してよい
3. **エラー時はスキップ** - 1ファイルの失敗が他に影響しないようにする
4. **モックが正とは限らない** - 明らかな誤りは仕様書を優先し、差分を報告
5. **変更しない情報** - 機能ID、優先度、ビジネスルール、技術仕様

## 実行例

```
/sync-mock-to-spec docs/screens/mock/v1
```
