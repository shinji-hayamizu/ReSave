# Claude Code 開発ガイド

このファイルは全プロジェクト共通の開発ガイドです。

### 基本フロー

**すべての作業は以下のフローに従うこと**

1. **依頼受付**: ユーザーから新しいタスク・依頼を受ける
2. **コード理解**: 必要な既存コードを読み込み、設計や処理を理解する
3. **情報収集**: さらに必要であればWebサーチなどで情報を収集する
4. **計画提示**: 修正内容や開発計画など実際の作業内容をテキストで提示する
5. **承認待機**: ユーザーからの明示的な承認を待つ
6. **段階的実行**: 承認後、1ステップずつ実行して完了報告する
7. **次ステップ**: 次のステップの承認を得て繰り返す

---

### 重要な原則

#### 計画段階でのファイル操作は絶対禁止

- **ステップ4（計画提示）では絶対にファイルの作成・編集・削除を行ってはならない**
- ユーザーが「具体的な内容を提示して」「修正内容を提示して」と言った場合、これは**ステップ4の計画提示を求めている**
- **ステップ6（段階的実行）に進むには、ユーザーが「実施して」「進めて」などと明示的に指示する必要がある**

#### 「新しいタスク・依頼」の定義（ステップ1）

以下の場合に基本フローを開始する:
- ユーザーが明示的に新しい作業を依頼した時（機能開発、修正、リファクタリング、テスト追加、ドキュメント更新、コミット作成など、種類を問わない）
- 大きなPhaseの開始時（Phase 1, Phase 2など）

以下は「新しいタスク」に含まない:
- 同一タスク内の継続作業（同じファイルの複数箇所修正など）

---

### 違反例（絶対禁止）

以下の行為は重大な規約違反である:

- **ステップ5をスキップ**: 計画提示なしにいきなり実装
- **ステップ6をスキップ**: 承認を待たずにファイル操作を実行
- 過去の記憶や推測に基づいて規約を判断
- 「提示して」と言われて勝手にファイルを作成・編集・削除
- 「内容を確認」と言われて勝手に実行
- ユーザーの明示的な実行指示なしでのファイル操作
- コミット作成時にDEVELOPMENT.mdを確認しない

**このルールの違反は重大な規約違反である**

## 【厳命】コミットメッセージ規約

**コミット作成時は必ずこの規約を遵守すること**

### フォーマット（1行のみ）

```
[<tag>] <type>(<scope>): <summary>
```

- `<tag>`（任意）: プロジェクト固有タグ（例: `AI-DEV`）
- `<type>`（任意）: `update`, `fix`, `docs`, `chore`
- `<scope>`（任意）: 影響範囲（例: `attendance`, `auth`, `test`）
- `<summary>`（必須）: 変更内容の要約（命令形、72文字程度、末尾ピリオド不要）

### 重要なルール

- **英語で記述**: コミットメッセージは英語で書くこと
- **1行のみ**: 複数行のメッセージは禁止
- **絵文字禁止**: コミットメッセージに絵文字を含めてはならない
- **命令形**: "add", "fix", "update" など（"added", "fixes"は不可）
- **簡潔に**: 72文字程度を目安

### 良い例

```
[AI-DEV] update(attendance): add timesheet validation
fix(auth): resolve token refresh issue
docs: update API specification
chore(docker): update MySQL to 8.0
```

### 悪い例

```
Update code                    # 何を変更したか不明
[AI-DEV] updated attendance... # 過去形
🤖 Generated with Claude       # 絵文字使用
[AI-DEV] docs: update ATTENDANCE.md

Generated with Claude Code     # 複数行
```

**このルールの違反は重大な規約違反である**

## 【厳命】ドキュメント更新の責務

**コード変更を行う場合、対応するドキュメントも必ず同時に更新すること**

ドキュメントとコードの乖離は重大な問題である。

### 基本原則

- コード変更に伴い、関連するドキュメント（`docs/`配下およびトップレベルのREADME.md、CLAUDE.md）の更新が必要かを必ず検討する
- 更新が必要と判断した場合は、コード変更と同時にドキュメントも更新する
- **更新が必要か判断を迷った場合は、必ずユーザーに確認すること**

### ドキュメント記述の原則

**ドキュメントは常に最新のコードに従った汎用的な情報のみを記述すること**

ステート情報（過去の変遷・履歴・移行経緯を知らないと理解できない情報）をドキュメントに含めてはならない。

**禁止される記述例:**
- 「v2.0構造」「v1.29以降」など特定バージョンへの言及
- 「旧書式」「新書式」「移行後」など変遷を前提とした表現
- 現在存在しないディレクトリ構造やファイル形式への参照
- 「以前は〜だった」「〜から変更された」など履歴情報

**理由**: ドキュメントを読む人は過去の変遷を知らない。常に「今」の状態のみを説明すること。

**例外**: コマンド例のバージョン引数（例：`./scripts/prompt-sync.sh v1.29`）は、実際に使用するバージョン名の例として許容。

### 作業手順

1. **コード変更前**: 該当ドキュメントを読み込み、現在の記載内容を把握
2. **コード変更実施**: 規約に従ってコードを修正
3. **ドキュメント影響判断**: 変更がドキュメントに影響するか検討
4. **更新または確認**: 影響がある場合は更新、迷った場合はユーザーに確認
5. **ユーザーに報告**: 実施内容を明示（「ドキュメント更新も完了」または「ドキュメント更新の要否を確認させてください」）

**違反例（絶対禁止）:**
- コードだけ変更してドキュメント更新を検討しない
- 迷ったまま自己判断でドキュメント更新をスキップする
- 「後でまとめて更新」と先送りする
- ステート情報をドキュメントに記述する

**このルールの違反は重大な規約違反である**

## 【厳命】スクリプト・テンポラルファイルの取り扱い

### Pythonスクリプトの使用禁止

**Pythonスクリプトの作成・実行は原則禁止**

- ユーザーから明示的に「Pythonで」「Pythonスクリプトで」と指示がない限り、Pythonファイルを作成してはならない
- 自動化処理が必要な場合は、bashコマンドまたはsedなどのUNIXコマンドを使用すること
- Pythonに限らず、勝手にスクリプト言語（Ruby、Perl等）のファイルを作成してはならない

### テンポラルファイルの即時削除

**一時的な処理ファイルは実行後すぐに削除すること**

- シェルスクリプト、変換スクリプトなど、一時的な処理のために作成したファイルは、実行完了後すぐに削除する
- プロジェクトのルートディレクトリや作業ディレクトリに処理ファイルを残してはならない
- 削除を忘れると、リポジトリに不要なファイルが残り、コミット対象になる可能性がある

**正しい手順:**
1. テンポラルファイルを作成（必要な場合のみ）
2. 実行
3. **即座に削除**

**違反例（絶対禁止）:**
- Pythonスクリプトを勝手に作成して実行
- テンポラルファイルを実行後に削除せず放置
- 「後で削除」として先送りする

**このルールの違反は重大な規約違反である**
