# Given-When-Thenパターン

本ドキュメントはGiven-When-Thenパターンの原則を説明します。
技術スタック固有の実装例は各技術スタックのドキュメントを参照してください。

## パターンの目的

- **可読性向上**: テストの意図が明確
- **保守性向上**: 構造が統一されている
- **デバッグ容易**: 失敗箇所の特定が簡単

---

## 基本構造

すべてのテストケースは以下の3段階構造に従う:

```
Given (準備):
  テスト対象の初期状態を構築

When (実行):
  テスト対象のメソッド・関数を実行

Then (検証):
  実行結果が期待通りか検証
```

**補足:** このパターンは「AAA (Arrange-Act-Assert) パターン」とも呼ばれます（Arrange=Given, Act=When, Assert=Then）。本プロジェクトでは統一のためGiven-When-Thenを使用します。

---

## Given (準備フェーズ)

### 目的
テスト対象の初期状態を構築する

### 含まれる内容
- テストデータの準備
- モックの設定
- 環境変数の設定
- 前提条件の構築

### 原則
- テストに必要な最小限のセットアップのみ
- 複雑なセットアップは別の準備関数に切り出す
- 他のテストに影響を与えない独立した準備

---

## When (実行フェーズ)

### 目的
テスト対象のメソッド・関数を実行する

### 含まれる内容
- テスト対象メソッドの呼び出し
- 結果の受け取り

### 原則
- 1つのテストで1つの実行のみ
- 実行結果は必ず変数に保存（Thenで検証するため）
- 副作用がある場合も検証可能な形で実行

---

## Then (検証フェーズ)

### 目的
実行結果が期待通りか検証する

### 含まれる内容
- 戻り値の検証
- 状態変化の検証
- 副作用の検証
- モック呼び出しの検証

### 原則
- 全ての重要な側面を検証
- 同一関心事内なら複数の検証が可能
- 検証失敗時のエラーメッセージが明確

---

## パターンの適用例（概念）

### 例1: 単純な関数のテスト

```
テスト: 時刻文字列を秒数に変換する

Given:
  - 入力: 時刻文字列 "09:30"

When:
  - timeToSeconds関数を実行

Then:
  - 結果が 34200 秒であることを検証
```

### 例2: データベース操作のテスト

```
テスト: ユーザーIDでユーザーを取得する

Given:
  - モックデータベース接続を準備
  - ユーザーID: 1
  - モックDB応答: { id: 1, name: "Test User" }

When:
  - findById(1) を実行

Then:
  - 返されたユーザーのIDが 1 であることを検証
  - 返されたユーザーの名前が "Test User" であることを検証
  - データベースへのクエリが正しく実行されたことを検証
```

### 例3: API呼び出しのテスト

```
テスト: ユーザー取得APIが正しいレスポンスを返す

Given:
  - ユーザーID: "user-123"
  - モックデータベースにユーザーが存在
  - HTTPリクエストを構築

When:
  - GET /api/users/user-123 を実行

Then:
  - HTTPステータスが 200 であることを検証
  - レスポンスボディにユーザー情報が含まれることを検証
  - レスポンス形式が正しいことを検証
```

## エラーケースのテスト

エラーが発生するケースでも同じパターンを適用:

```
テスト: 存在しないユーザーIDで取得を試みる

Given:
  - モックデータベース接続を準備
  - 存在しないユーザーID: 999
  - モックDB応答: 空の結果

When:
  - findById(999) を実行

Then:
  - 結果が null であることを検証
  または
  - エラーがthrowされることを検証
```

### エラーケーステストの考え方

**原則: 実装されているエラーハンドリングを全てテストする**

エラーケーステストを網羅するには、以下の手順で考える：

1. **テスト対象のコードを読む**
   - if文、try-catch、early returnなど、全ての分岐を確認
   - 各分岐でどのようなエラーが想定されているか理解

2. **各エラーハンドリングに対してテストを作成**
   - コードに書かれている分岐ごとにテストケースを作成
   - 正常系だけでなく、全てのelse、catch、エラーreturnをカバー

3. **カバレッジで確認**
   - テスト実行後、カバレッジレポートで未カバー分岐を確認
   - 未カバー箇所に対応するテストを追加

**重要:** エラーケースを事前に列挙するのではなく、**実装を基準に**テストケースを決定する。

### 例：エラーハンドリングから逆算する

```
実装コード（概念）:
  function processData(input) {
    if (!input) {              // 分岐1: nullチェック
      return null;
    }

    if (input.length > 100) {  // 分岐2: 範囲チェック
      throw new Error('Too long');
    }

    try {
      return transform(input);  // 正常系
    } catch (error) {           // 分岐3: 変換エラー
      throw new Error('Transform failed');
    }
  }

必要なテストケース:
  - 正常系: 有効な入力 → 変換された結果を返す
  - 分岐1: null/undefined → nullを返す
  - 分岐2: 101文字の入力 → 'Too long'エラーをthrow
  - 分岐3: 変換失敗 → 'Transform failed'エラーをthrow
```

このように、**実装の分岐からテストケースを導出**する。

## 非同期処理のテスト

非同期処理でも同じパターンを適用:

```
テスト: 非同期でデータを取得する

Given:
  - モックAPIエンドポイントを準備
  - 非同期応答データを設定

When:
  - fetchData関数を非同期実行
  - 完了を待機

Then:
  - 取得したデータが期待通りであることを検証
```

---

## パターン適用のベストプラクティス

### 1. 各フェーズを明確に分離

- Given/When/Thenの間に空行を入れる
- コメントで各フェーズを明示する
- フェーズが混在しないようにする

### 2. Given は簡潔に

- 必要最小限のセットアップのみ
- 複雑な準備は別関数に切り出す
- テストごとに独立した準備を行う

### 3. When は1つの実行のみ

- 1つのテストで複数の実行を行わない
- 異なる実行は別のテストに分割

### 4. Then は包括的に

- 重要な側面を全て検証
- 同一関心事内なら複数検証OK
- 検証漏れがないようにする

## よくある間違い

### 間違い1: フェーズの混在

**Bad:**
```
テスト実装:
  データを準備
  関数を実行
  追加データを準備  (Givenに戻っている)
  別の関数を実行    (複数のWhen)
  結果を検証
```

**Good:**
```
テスト1:
  Given: データを準備
  When:  関数を実行
  Then:  結果を検証

テスト2:
  Given: 追加データを準備
  When:  別の関数を実行
  Then:  結果を検証
```

### 間違い2: Given での検証

**Bad:**
```
Given:
  データを準備
  データが正しいか検証 (Givenでは検証しない)
```

**Good:**
```
Given:
  データを準備（検証なし）

When:
  関数を実行

Then:
  結果を検証（ここで検証）
```
