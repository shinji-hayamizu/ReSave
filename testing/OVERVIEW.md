# テスト原則の全体像

本ドキュメントはバリアフリー雇用支援プラットフォームにおけるテスト原則の全体像を定義します。
これらの原則は全ての技術スタックに共通して適用されます。

## 最重要原則

### 1. テストカバレッジ100%厳守

- すべてのコードにテストが必要
- カバレッジ100%未達のコミットは禁止
- 例外は認めない

理由: どんなに確信があっても、実際のテストなしでは予期せぬバグが混入する。

詳細: [COVERAGE.md](COVERAGE.md)

### 2. Given-When-Thenパターン厳守

- すべてのテストケースはこのパターンに従う
- テストの可読性と保守性を確保

```
テスト構造:
  Given: テスト対象の初期状態を準備
  When:  テスト対象の処理を実行
  Then:  結果を検証
```

詳細: [PATTERNS.md](PATTERNS.md)

### 3. 1テスト1関心事の原則

- 各テストは1つの論理的関心事（テストシナリオ）を検証する
- 同一関心事内の複数側面は同時検証可
- 異なる関心事は別テストケースに分割する

詳細: [ONE_CONCERN.md](ONE_CONCERN.md)

### 4. 実動作確認必須

- 実装完了後は必ず実際の動作確認
- ユニットテストだけでは不十分
- 統合テスト・E2Eテストも実施

---

## テストレベル

### ユニットテスト (Unit Tests)

**対象:**
- ユーティリティ関数
- バリデーション関数
- リポジトリメソッド
- ハンドラー関数

**特徴:**
- 外部依存を完全にモック化
- 高速実行
- 独立して実行可能

### 統合テスト (Integration Tests)

**対象:**
- API Routes
- データベーストランザクション
- 複数モジュール間の連携

**特徴:**
- 主要な外部サービスのみモック
- エンドツーエンドのデータフロー検証

### コンポーネントテスト (Component Tests)

**対象:**
- UIコンポーネント
- UIインタラクション
- アクセシビリティ

**特徴:**
- ブラウザDOMシミュレーション環境
- ユーザー視点のテスト

---

## モック戦略

外部依存は必ずモック化:

- データベース接続
- 外部API・認証サービス
- キャッシュ（Redis等）
- HTTP通信
- ファイルシステム

**理由:**
1. **テスト速度**: DBやAPI通信がないため高速
2. **テスト安定性**: 外部サービスの状態に依存しない
3. **テスト独立性**: テスト間で状態が共有されない
4. **エラーケースのテスト**: 意図的にエラーを発生させられる

---

## 命名規則

### テストケース名の基本形式

**形式:** `条件: 期待される動作`

**例:**
- 無効な時刻文字列("25:00"): falseを返す
- ユーザーが存在する場合: ユーザーを返す
- データベースエラー: エラーをthrowする

詳細: [NAMING.md](NAMING.md)

---

## 禁止事項

1. **テストなしのコミット** - 絶対禁止
2. **異なる関心事を1テストに含める** - 可読性・保守性の低下
3. **テスト間の依存関係** - 独立性の破壊
4. **ハードコードされたテストデータ** - 保守性の低下
5. **実データベース・外部APIの使用** - 安定性の破壊
6. **曖昧なテスト名** - 可読性の低下
