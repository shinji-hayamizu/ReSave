# 1テスト1関心事の原則

本ドキュメントは「1テスト1関心事」の原則を詳しく説明します。
技術スタック固有の実装例は各技術スタックのドキュメントを参照してください。

## 原則の定義

**各テストは1つの論理的関心事（テストシナリオ）を検証する**

- 同一関心事内の複数側面は同時検証可（複数の検証あり）
- 異なる関心事は別テストケースに分割する

---

## 「関心事」とは

### 関心事の定義

**関心事 = 検証したい1つの論理的なシナリオ**

- ユーザーを作成できるか
- 無効な入力でエラーが返されるか
- API呼び出しが正しいレスポンスを返すか

### 関心事 ≠ 検証の数

1つの関心事を検証するために複数の検証（assertion）を行うことは許容される。

---

## 同一関心事とみなせる例

### 1. オブジェクトの複数プロパティが正しく設定されているか

```
関心事: ユーザーオブジェクトが正しく作成される

検証:
- ID が設定されている
- 名前が正しい
- メールアドレスが正しい
- 作成日時が設定されている

→ 全て「ユーザーオブジェクトの構造」という同一関心事
```

### 2. APIレスポンスのステータスコードとボディが正しいか

```
関心事: APIが正しいレスポンスを返す

検証:
- ステータスコードが 200
- レスポンスボディに successフラグ
- レスポンスボディにデータが含まれる

→ 全て「APIレスポンス」という同一関心事
```

### 3. 関数実行後の複数の状態変化が正しいか

```
関心事: トランザクションが正しくコミットされる

検証:
- データベースにレコードが保存されている
- commitが呼ばれた
- rollbackが呼ばれていない

→ 全て「トランザクション成功」という同一関心事
```

## 異なる関心事とみなすべき例

### 1. 正常系と異常系

```
Bad（1つのテストに混在）:
  ユーザー作成のテスト
    - 正常なデータ: 成功
    - 無効なデータ: エラー

Good（分割）:
  テスト1: 正常なデータでユーザー作成成功
  テスト2: 無効なデータでバリデーションエラー
```

### 2. 作成処理と取得処理

```
Bad（1つのテストに混在）:
  ユーザー処理のテスト
    - ユーザーを作成
    - 作成したユーザーを取得

Good（分割）:
  テスト1: ユーザー作成成功
  テスト2: ユーザー取得成功
```

### 3. 異なる入力パターンに対する挙動

```
Bad（1つのテストに混在）:
  時刻検証のテスト
    - "09:30": true
    - "25:00": false
    - null: false

Good（分割）:
  テスト1: 正常な時刻文字列("09:30"): trueを返す
  テスト2: 無効な時刻文字列("25:00"): falseを返す
  テスト3: nullが渡された場合: falseを返す
```

---

## 異なる関心事の分割例

### Bad: 複数の関心事を1テストに含める

```
テスト: ユーザー処理

処理:
  1. ユーザーを作成
  2. 作成したユーザーを取得
  3. ユーザーを削除

問題点:
- 3つの異なる関心事が混在
- どれかが失敗した時、原因特定が困難
- テストの独立性が失われる
```

### Good: 関心事ごとに分割

```
テスト1: ユーザー作成成功
  Given: ユーザーデータ
  When:  create を実行
  Then:  ユーザーが作成される

テスト2: ユーザー取得成功
  Given: ユーザーIDが存在
  When:  findById を実行
  Then:  ユーザーが返される

テスト3: ユーザー削除成功
  Given: ユーザーIDが存在
  When:  delete を実行
  Then:  ユーザーが削除される
```

---

## 原則の利点

### 1. テスト失敗時の原因特定が容易

**Bad（複数関心事）:**
```
テスト失敗: "ユーザー処理"
→ 作成が失敗？取得が失敗？削除が失敗？
```

**Good（1関心事）:**
```
テスト失敗: "ユーザー取得成功: ユーザーを返す"
→ 取得処理に問題があると即座に判明
```

### 2. テストの独立性

各テストが独立して実行可能:
- 他のテストに影響しない
- 並列実行が可能
- テストの順序に依存しない

### 3. 柔軟な検証

同一関心事内では複数の検証が可能:
- オブジェクトの全プロパティを検証
- APIレスポンスの全側面を検証
- 必要な全ての状態変化を検証

### 4. 保守性

- テストの追加・削除が容易
- 関心事が明確なので修正箇所が分かりやすい
- リファクタリング時の影響範囲が限定的

## よくある質問

### Q1: 複数の検証は1テスト1関心事に違反しないか？

**A:** 違反しない。同一関心事内の複数側面を検証することは推奨される。

```
Good: 1つの関心事、複数の検証
  関心事: ユーザーオブジェクトが正しい
  検証: ID、名前、メール、作成日時

Bad: 複数の関心事
  関心事1: ユーザー作成
  関心事2: ユーザー取得
  → 別々のテストに分割すべき
```

### Q2: どこまで分割すべきか？

**A:** 異なるシナリオは必ず分割、同じシナリオの側面は統合。

```
適切な分割:
  - 正常な時刻("09:30"): true
  - 無効な時刻("25:00"): false
  - null: false

過剰な分割:
  - ユーザーのIDが正しい
  - ユーザーの名前が正しい
  - ユーザーのメールが正しい
  → 「ユーザーオブジェクトが正しい」として統合すべき
```

### Q3: 準備処理が重複する場合は？

**A:** 共通の準備処理は beforeEach 等で共有できる。関心事の分割とは独立。

```
全テストで共通:
  beforeEach: データベース接続のモック準備

テスト1: ユーザー作成成功
  Given: ユーザーデータ
  When:  create
  Then:  成功

テスト2: ユーザー取得成功
  Given: ユーザーID
  When:  findById
  Then:  成功
```

## まとめ

### 1テスト1関心事の原則

- **定義**: 各テストは1つの論理的シナリオを検証
- **複数検証OK**: 同一関心事なら複数の検証が可能
- **分割必須**: 異なる関心事は必ず別テストに分割

### 判断基準

**同一関心事:**
- オブジェクトの複数プロパティ
- APIレスポンスの全側面
- 関連する複数の状態変化

**異なる関心事:**
- 正常系 vs 異常系
- 作成 vs 取得 vs 削除
- 異なる入力パターン
