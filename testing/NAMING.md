# 命名規則

本ドキュメントはテストコードにおける命名規則の原則を説明します。
技術スタック固有の実装例は各技術スタックのドキュメントを参照してください。

## 基本原則

### 明確さ
- 曖昧な名前を避ける
- 一目でテスト内容がわかる名前をつける

### 一貫性
- プロジェクト全体で統一された命名
- チームメンバー全員が同じルールに従う

### 具体性
- 条件と期待動作を明示する
- 抽象的な名前を避ける

---

## テストファイル名

### 原則

**形式:** `{対象ファイル名}.test.{拡張子}`

### 例（概念）

```
対象ファイル              テストファイル
-----------------        --------------------
lib/time              →  lib/time.test
lib/validation        →  lib/validation.test
handlers/validate     →  handlers/validate.test
components/UserInfo   →  components/UserInfo.test
```

### ルール

- 対象ファイル名と同じ名前
- `.test.` を挿入
- 拡張子は対象ファイルと同じ
- ディレクトリ構造も対象ファイルと同じ

---

## テストスイート名（グループ名）

### 第1階層: モジュール名またはコンポーネント名

対象ファイルのエクスポートされている主要なモジュール・コンポーネント名を使用

**例:**
```
対象: lib/validation     → テストスイート: validation
対象: components/UserInfo → テストスイート: UserInfo
対象: repositories/user   → テストスイート: UserRepository
```

### 第2階層: メソッド名または機能名

対象メソッド・関数名を使用

**例:**
```
validation
  ├─ escapeHtml
  ├─ sanitizeInput
  └─ validateTimeString

UserRepository
  ├─ findById
  ├─ create
  └─ update
```

## テストケース名

### 形式

**基本形式:** `条件: 期待される動作`

### 構成要素

1. **条件**: テスト対象への入力や前提条件
2. **期待される動作**: 期待される結果や挙動

### 原則

- **具体的な条件を明示**: 「無効な入力」ではなく「無効な時刻文字列("25:00")」
- **期待される動作を明確に**: 「動作する」ではなく「falseを返す」
- **日本語推奨**: 本プロジェクトでは日本語を使用（技術スタックにより異なる可能性あり）

---

## テストケース名の例

### Good例（具体的で明確）

```
無効な時刻文字列("25:00"): falseを返す
正常な時刻文字列("09:30"): trueを返す
nullが渡された場合: 空文字列を返す
ログアウトボタンクリック: onLogout関数が呼ばれる
ユーザー作成成功: 正しいユーザーオブジェクトを返す
データベースエラー: エラーをthrowする
配列が空の場合: 空の結果を返す
権限がない場合: 403エラーを返す
```

### Bad例（曖昧・不明確）

```
時刻検証が動作する
  → 何を検証？どんな入力？何を期待？

エラーケース
  → どんなエラー？何を検証？

正常系
  → どんな正常系？何を検証？

test1
  → 何をテスト？

works
  → 何が動作？

falseを返す
  → どんな条件で？

無効な時刻文字列
  → 何を期待？
```

---

## 条件の書き方

### 具体的な値を含める

**Good:**
- 無効な時刻文字列("25:00"): ...
- ユーザーID(123)が存在する場合: ...
- 配列が空([])の場合: ...

**Bad:**
- 無効な時刻文字列: ...
- ユーザーが存在する場合: ...
- 配列が空の場合: ...（これは許容されるが、より具体的な方が良い）

### 前提条件を明示

**Good:**
- ログイン済みユーザー: ダッシュボードを表示する
- 認証トークンが期限切れの場合: 401エラーを返す
- データベース接続エラー: リトライ処理を実行する

---

## 期待される動作の書き方

### 具体的な結果を記述

**Good:**
- ...を返す
- ...をthrowする
- ...が呼ばれる
- ...が表示される
- ...にリダイレクトする
- ...に保存される

**Bad:**
- ...が動作する
- ...が成功する
- ...が処理される

### 複数の検証を含む場合

同一関心事内で複数の側面を検証する場合、主要な検証を記述:

**Good:**
```
ユーザー作成成功: 正しいユーザーオブジェクトを返す
（テスト内でID、名前、メールなど複数のプロパティを検証）
```

## 変数名

### テストデータ

**原則:** 型や用途を示す名前

**Good例:**
```
userId = 'user-123'
mockUser = { id: userId, name: 'Test User' }
invalidEmail = 'invalid-email'
emptyString = ''
nullValue = null
expectedResult = 'processed-data'
```

**Bad例:**
```
data = 'user-123'          (何のデータ？)
obj = { id: 1 }            (何のオブジェクト？)
x = 'test'                 (意味不明)
```

### モックオブジェクト

**形式:** `mock{対象名}`

**例:**
```
mockConnection
mockFetch
mockDatabase
mockRedisClient
mockAuthService
```

## 言語別の考慮事項

各プログラミング言語には固有の命名慣習があります:

### 一般的なパターン

- **英語圏のプロジェクト**: テストケース名も英語
- **日本語プロジェクト**: テストケース名は日本語（本プロジェクト）
- **多言語プロジェクト**: コードは英語、コメント・テスト名は各言語

### 技術スタック固有の命名

具体的な命名規則は各技術スタックのドキュメントを参照:

| 技術スタック | ドキュメント |
|-------------|-------------|
| TypeScript/JavaScript | [typescript/GUIDELINES.md](typescript/GUIDELINES.md) |

## テストスイート全体の例（概念）

```
UserRepository
  ├─ findById
  │   ├─ ユーザーが存在する場合: ユーザーを返す
  │   ├─ ユーザーが存在しない場合: nullを返す
  │   └─ データベースエラー: エラーをthrowする
  │
  ├─ create
  │   ├─ ユーザー作成成功: 正しいユーザーオブジェクトを返す
  │   ├─ 必須フィールドが欠けている場合: バリデーションエラーを返す
  │   └─ 重複するメールアドレス: エラーをthrowする
  │
  └─ update
      ├─ ユーザー更新成功: 更新されたユーザーを返す
      ├─ ユーザーが存在しない場合: nullを返す
      └─ 無効なデータ: バリデーションエラーを返す
```

## まとめ

### 命名規則の利点

1. **可読性**: 一目でテスト内容がわかる
2. **保守性**: ファイル・テストの検索が容易
3. **一貫性**: チーム全体で統一された命名
4. **デバッグ**: テスト失敗時の原因特定が容易

### 基本原則

- **明確さ**: 曖昧な名前を避ける
- **一貫性**: プロジェクト全体で統一
- **具体性**: 条件と期待動作を明示
