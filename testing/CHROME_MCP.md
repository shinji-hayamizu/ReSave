# Chrome MCP統合テスト

本ドキュメントはAIがChrome MCPを使用して実施する統合テストの要件定義です。

## 目的

実装したUI機能が実際のブラウザ環境で正しく動作することを検証する。

---

## 前提

- **AIがChrome MCPツールを使用してブラウザ統合テストを実施する**
- テストは実装完了後、コミット前に必ず実行する
- テスト結果はスクリーンショットとログで報告する

---

## 実施タイミング

以下の場合に**必ず実施**:

- **コンポーネント実装完了後**
- **API実装完了後**
- **コミット前**

---

## 検証項目

AIは以下の項目を必ず検証すること:

### 1. キーボード操作

すべてのインタラクティブ要素がキーボードでアクセス可能であることを確認:

- **Tab/Shift+Tab**: フォーカス移動
- **Enter**: フォーム送信・決定
- **Escape**: キャンセル・閉じる
- **Arrow keys**: 選択・移動
- **Space**: チェックボックス切替

### 2. フォーム操作

フォームが正しく機能することを確認:

- 入力値のバリデーション
- エラーメッセージ表示
- 送信成功時の挙動
- **ローディング状態の表示**

### 3. アクセシビリティ

**WCAG 2.1 AA準拠**を確認:

- **ARIA属性の確認**
  - `role`
  - `aria-label`
  - `aria-labelledby`
  - `aria-describedby`
  - `aria-invalid`
- フォーカス状態の視覚化
- スクリーンリーダー対応
- 十分な色コントラスト

### 4. 認証フロー

- 未認証時に適切にログインページへリダイレクトされるか
- 認証後に保護されたページにアクセスできるか
- ログアウト後に保護されたページにアクセスできないか

### 5. レスポンシブ動作

- モバイル画面サイズで表示が崩れないか
- タッチ操作が適切に動作するか（該当する場合）

### 6. エラーハンドリング

- ネットワークエラー時に適切なメッセージが表示されるか
- バリデーションエラーが分かりやすく表示されるか
- エラー発生後に正常な状態に復帰できるか

---

## アクセシビリティチェックリスト

AIは以下のチェックリストを使用して検証すること:

### フォーカス管理

- [ ] すべてのインタラクティブ要素がTabでアクセス可能
- [ ] フォーカス順序が論理的
- [ ] フォーカス状態が視覚的に明確
- [ ] フォーカストラップが適切に機能（モーダル等）

### ARIA属性

- [ ] 適切なrole属性が設定されている
- [ ] aria-labelまたはaria-labelledbyが設定されている
- [ ] aria-describedbyで説明テキストが関連付けられている
- [ ] aria-invalidでエラー状態が示されている

### キーボード操作

- [ ] Enterで送信・決定ができる
- [ ] Escapeでキャンセル・閉じることができる
- [ ] 矢印キーで選択・移動ができる（該当する場合）
- [ ] スペースキーでチェックボックス切替ができる

### エラーハンドリング

- [ ] エラーメッセージが明確
- [ ] エラー箇所が視覚的に示されている
- [ ] エラー箇所にフォーカスが移動する
- [ ] エラー解消後、正常に進める

---

## 必須テスト項目

### 全機能共通（必ず実施）

1. **初期表示確認**
   - ページが正しく読み込まれること
   - 必要な要素が全て表示されること

2. **認証状態確認**
   - 未認証時の動作
   - 認証済み時の動作

3. **正常系フロー**
   - 主要な操作が正常に完了すること
   - 結果が正しく表示されること

4. **異常系フロー**
   - 無効な入力でエラーメッセージが表示されること
   - エラー後に再操作可能であること

5. **キーボード操作**
   - Tab/Shift+Tabでフォーカス移動
   - Enterで送信・決定

6. **アクセシビリティチェックリスト**
   - 上記16項目をすべて確認

### フォーム機能（該当する場合）

7. **バリデーション**
   - 必須項目の未入力チェック
   - 形式不正のチェック
   - エラーメッセージの表示位置と内容

8. **送信処理**
   - 送信中の状態表示（ローディング等）
   - 成功時の画面遷移またはメッセージ
   - 失敗時のエラー表示

### データ表示機能（該当する場合）

9. **データ取得**
   - データが正しく表示されること
   - 空データ時の表示
   - 読み込みエラー時の表示

## AIテスト実行の検証基準

人間が以下の基準でAIのテスト実行を検証する:

### 1. テスト網羅性
- 上記「必須テスト項目」が全て実施されているか
- 該当する「検証項目」が確認されているか
- 「アクセシビリティチェックリスト」16項目が確認されているか

### 2. 証跡の妥当性
- スクリーンショットで動作が確認できるか
- エラーが発生した場合、ログが提示されているか
- 各テストケースの期待結果と実際の結果が明示されているか

### 3. 問題の報告
- 不具合を発見した場合、明確に報告されているか
- 再現手順が記載されているか
- エラーメッセージやスタックトレースが提示されているか

### 報告時の注意事項

**AIがテストを実施する場合:**
1. 上記テンプレートに従ってレポートを作成
2. 対話内で提示
3. ユーザーの承認を待つ
4. 承認後にコミット可能

**人間がテストを実施する場合:**
1. 実際にブラウザで動作確認
2. 上記チェックリストを確認
3. 問題がなければコミット

### コミット前チェックリスト

コミット前に以下を確認してください：

- [ ] Chrome MCPテスト（またはブラウザ実動作確認）を実施した
- [ ] すべてのテスト項目がPassしている
- [ ] 発見した問題を修正した（または記録した）

**重要:** テスト失敗時は絶対にコミットしない

---

## テスト失敗時の対応

AIがテスト失敗を検出した場合:

1. **即座に報告**: コミット前に必ずユーザーに報告
2. **原因分析**: 実装の問題かテスト環境の問題かを判断
3. **修正提案**: 実装の問題であれば修正案を提示
4. **再テスト**: 修正後に再度テストを実施

**重要**: テスト失敗時は絶対にコミットしない

---

## まとめ

### Chrome MCP統合テストの利点

1. **実環境での動作確認**: 実際のブラウザでの挙動を検証
2. **アクセシビリティ検証**: WCAG 2.1 AA準拠とキーボード操作を確認
3. **バグの早期発見**: ユニットテストで見逃した問題を発見
4. **ユーザー体験の確認**: 実際の操作フローを検証

### 注意点

1. **ユニットテストとの組み合わせ**: Chrome MCPテストだけでは不十分
2. **環境依存**: ネットワーク遅延等の影響を受ける可能性
3. **定期的な実施**: コミット前には必ず実施
