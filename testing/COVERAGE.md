# カバレッジ原則

本ドキュメントはテストカバレッジの原則を説明します。
技術スタック固有の設定方法は各技術スタックのドキュメントを参照してください。

## 基本原則

### 100%厳守

以下のメトリクスすべてで**100%を達成**する必要があります。例外は認めません。

- **Lines (行カバレッジ)**: 100%
- **Functions (関数カバレッジ)**: 100%
- **Branches (分岐カバレッジ)**: 100%
- **Statements (文カバレッジ)**: 100%

### 理由

1. **バグ検出**: すべてのコードパスがテストされている
2. **リファクタリング安全性**: 変更時の影響を即座に検知
3. **ドキュメント**: テストが実装の仕様書になる
4. **品質保証**: コードベース全体の品質が保証される

---

## カバレッジメトリクスの意味

### Lines (行カバレッジ)

実行された行の割合

```
関数:
  行1: const x = 10;     (実行された)
  行2: if (x > 5) {      (実行された)
  行3:   return true;    (実行された)
  行4: }
  行5: return false;     (実行されていない)

カバレッジ: 75% (3/4行)
```

### Functions (関数カバレッジ)

実行された関数の割合

```
ファイル内の関数:
  - functionA()  (テストで呼ばれた)
  - functionB()  (テストで呼ばれた)
  - functionC()  (テストで呼ばれていない)

カバレッジ: 66.7% (2/3関数)
```

### Branches (分岐カバレッジ)

実行された分岐の割合

```
条件分岐:
  if (condition) {
    // 分岐A  (テストされた)
  } else {
    // 分岐B  (テストされていない)
  }

カバレッジ: 50% (1/2分岐)
```

### Statements (文カバレッジ)

実行された文の割合（Linesと似ているが、複数行にまたがる文を1つとカウント）

---

## 除外対象

以下は一般的にカバレッジ計算から除外されます（技術スタックにより異なる）:

- テストファイル自体
- モックファイル
- 設定ファイル
- 型定義ファイル
- ビルド出力

具体的な除外設定は各技術スタックのドキュメントを参照。

## カバレッジ確認の基本フロー

### 1. カバレッジレポート生成

テストを実行してカバレッジレポートを生成

### 2. 結果の確認

全メトリクスが100%であることを確認

### 3. 未カバー箇所の特定

100%未達の場合、未カバーの箇所を特定

### 4. 追加テストケース作成

未カバー箇所に対してテストケースを追加

### 5. 再度確認

100%達成まで繰り返し

---

## よくある未カバー箇所

### 1. エラーハンドリングの分岐

```
処理:
  try {
    正常処理
  } catch (error) {
    エラー処理  (このブロックがカバーされていない)
  }
```

**対処**: エラーが発生するケースのテストを追加

### 2. 条件分岐の一部

```
関数:
  if (value > 100) {
    // 大きい値の処理  (この分岐がカバーされていない)
  }
  return 通常処理;
```

**対処**: 条件が真になるケースのテストを追加

### 3. デフォルト値の分岐

```
関数:
  return name || 'デフォルト名';
         (デフォルト値の分岐がカバーされていない)
```

**対処**: nameが空の場合のテストを追加

### 4. 早期リターンのパターン

```
関数:
  if (!data) {
    return デフォルト結果;  (この早期リターンがカバーされていない)
  }
  return 通常処理(data);
```

**対処**: dataがnullの場合のテストを追加

## カバレッジ100%の注意点

### カバレッジ100% ≠ バグゼロ

カバレッジ100%は「すべてのコードが実行された」ことを示すだけで、「ロジックが正しい」ことは保証しません。

**例:**

```
関数: 2つの数を足す
  return a - b;  (間違っているが、テストが実行されればカバレッジ100%)

テスト:
  Given: a = 5, b = 3
  When:  add(a, b) を実行
  Then:  結果が 2 であることを検証  (このテストは通る（誤実装を検出できない）)
```

**対策**:
- 期待値を正しく設定する
- エッジケースをテストする
- 複数の入力パターンでテストする

### 意味のあるテストを書く

カバレッジのためだけのテストは避ける:

**Bad:**
```
テスト: 関数を呼ぶだけ
  Given: データを準備
  When:  関数を実行
  Then:  （検証なし）
```

**Good:**
```
テスト: 関数が正しい結果を返す
  Given: データを準備
  When:  関数を実行
  Then:  結果が期待通りであることを検証
```

## テストカバレッジのベストプラクティス

### 1. 正常系と異常系を両方テスト

- 成功ケース: 正しい入力での動作
- 失敗ケース: 無効な入力、エラー発生時の動作

### 2. エッジケースを含める

- 境界値: 最大値、最小値、ゼロ
- 空の値: null, undefined, 空文字列、空配列
- 特殊なケース: 日をまたぐ時刻、うるう年など

### 3. 全ての分岐をカバー

- if-else の両方
- switch文の全てのcase
- 三項演算子の両方
- 論理演算子（&&, ||）の短絡評価

### 4. エラーハンドリングをテスト

- try-catch のcatchブロック
- エラー時の戻り値
- エラーメッセージの内容

## まとめ

### カバレッジ100%を達成するには

1. 全ての関数をテストする
2. 全ての分岐をテストする
3. エラーケースをテストする
4. エッジケースをテストする
5. カバレッジレポートで未カバー箇所を確認
6. 追加テストを作成して100%達成

### 注意事項

- カバレッジ100%は最低限の品質基準
- ロジックの正しさは別途検証が必要
- 意味のあるテストを書くことが重要
