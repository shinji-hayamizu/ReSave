
















## 【厳命】AI開発の作業進行規則

### 基本フロー

**すべての作業は以下のフローに従うこと**

1. **依頼受付**: ユーザーから新しいタスク・依頼を受ける
2. **ドキュメント確認**: 必要なドキュメントを読み込む（後述）
3. **コード理解**: 必要な既存コードを読み込み、設計や処理を理解する
4. **情報収集**: さらに必要であればWebサーチなどで情報を収集するつの
5. **計画提示**: 修正内容や開発計画など実際の作業内容をテキストで提示する
6. **承認待機**: ユーザーからの明示的な承認を待つ
7. **段階的実行**: 承認後、1ステップずつ実行して完了報告する
8. **次ステップ**: 次のステップの承認を得て繰り返す

---

### 重要な原則

#### 計画段階でのファイル操作は絶対禁止

- **ステップ5（計画提示）では絶対にファイルの作成・編集・削除を行ってはならない**
- ユーザーが「具体的な内容を提示して」「修正内容を提示して」と言った場合、これは**ステップ5の計画提示を求めている**
- **ステップ7（段階的実行）に進むには、ユーザーが「実施して」「進めて」などと明示的に指示する必要がある**

#### ドキュメント確認の詳細（ステップ2）

**必要なドキュメントは必ず読み込むこと**

- 同一セッション内で読み込み済みであることが確実な場合のみ、再読み込み不要
- **読み込み済みか不明な場合は、必ず読み込むこと**（auto-compactなどでコンテキスト圧縮された可能性を考慮）

**全タスク共通の必須読み込み:**
- `share-docs/CODING_STANDARDS.md` - コーディング規約
- `share-docs/DEVELOPMENT.md` - 開発ガイド（環境操作、テスト、コミット規約）

**タスク別必須読み込み（該当タスク実施時は必ず読み込むこと）:**
以下のタスクを実施する場合、対応するドキュメントを必ず読み込むこと。**読み飛ばしは重大な規約違反**。


- **API開発・修正時**: `docs/API.md`
- **マイグレーション作成・実行時**: `docs/MIGRATION.md`
- **デプロイ作業時**: `docs/DEPLOYMENT.md`
- **サービス固有機能の開発・修正時**: `docs/services/` 配下の該当サービスドキュメント
- **アーキテクチャ理解が必要な開発時**: `docs/ARCHITECTURE.md`
<!-- 追加 -->
- **テスト実装時*: `share-docs/TESTING.md`＞

**原則: 判断に迷った場合は、関連しそうなドキュメントをすべて読み込むこと**

#### 「新しいタスク・依頼」の定義（ステップ1）

以下の場合に基本フローを開始する:
- ユーザーが明示的に新しい作業を依頼した時（機能開発、修正、リファクタリング、テスト追加、ドキュメント更新、コミット作成など、種類を問わない）
- 大きなPhaseの開始時（Phase 1, Phase 2など）

以下は「新しいタスク」に含まない:
- 同一タスク内の継続作業（同じファイルの複数箇所修正など）

---

### 違反例（絶対禁止）

以下の行為は重大な規約違反である:

- **ステップ2をスキップ**: ドキュメントを読まずに作業を開始
- **ステップ5をスキップ**: 計画提示なしにいきなり実装
- **ステップ6をスキップ**: 承認を待たずにファイル操作を実行
- 過去の記憶や推測に基づいて規約を判断
- 「提示して」と言われて勝手にファイルを作成・編集・削除
- 「内容を確認」と言われて勝手に実行
- ユーザーの明示的な実行指示なしでのファイル操作
- コミット作成時にDEVELOPMENT.mdを確認しない

**このルールの違反は重大な規約違反である**

## 【厳命】コミットメッセージ規約

**コミット作成時は必ずこの規約を遵守すること**

### フォーマット（1行のみ）

```
[<tag>] <type>(<scope>): <summary>
```

- `<tag>`（任意）: プロジェクト固有タグ（例: `AI-DEV`）
- `<type>`（任意）: `update`, `fix`, `docs`, `chore`
- `<scope>`（任意）: 影響範囲（例: `attendance`, `auth`, `test`）
- `<summary>`（必須）: 変更内容の要約（命令形、72文字程度、末尾ピリオド不要）

### 重要なルール

- **英語で記述**: コミットメッセージは英語で書くこと
- **1行のみ**: 複数行のメッセージは禁止
- **絵文字禁止**: コミットメッセージに絵文字を含めてはならない
- **命令形**: "add", "fix", "update" など（"added", "fixes"は不可）
- **簡潔に**: 72文字程度を目安

### 良い例

```
[AI-DEV] update(attendance): add timesheet validation
fix(auth): resolve token refresh issue
docs: update API specification
chore(docker): update MySQL to 8.0
```

### 悪い例

```
Update code                    # 何を変更したか不明
[AI-DEV] updated attendance... # 過去形
🤖 Generated with Claude       # 絵文字使用
[AI-DEV] docs: update ATTENDANCE.md

Generated with Claude Code     # 複数行
```

**詳細**: [docs/DEVELOPMENT.md](docs/DEVELOPMENT.md)

**このルールの違反は重大な規約違反である**

## 【厳命】ドキュメント更新の責務

**コード変更を行う場合、対応するドキュメントも必ず同時に更新すること**

ドキュメントとコードの乖離は重大な問題である。

### 基本原則

- コード変更に伴い、関連するドキュメント（`docs/`配下、およびトップレベルのREADME.md、CLAUDE.md）の更新が必要かを必ず検討する
- 更新が必要と判断した場合は、コード変更と同時にドキュメントも更新する
- **更新が必要か判断を迷った場合は、必ずユーザーに確認すること**

### 作業手順

1. **コード変更前**: 該当ドキュメントを読み込み、現在の記載内容を把握
2. **コード変更実施**: 規約に従ってコードを修正
3. **ドキュメント影響判断**: 変更がドキュメントに影響するか検討
4. **更新または確認**: 影響がある場合は更新、迷った場合はユーザーに確認
5. **ユーザーに報告**: 実施内容を明示（「ドキュメント更新も完了」または「ドキュメント更新の要否を確認させてください」）

**違反例（絶対禁止）:**
- コードだけ変更してドキュメント更新を検討しない
- 迷ったまま自己判断でドキュメント更新をスキップする
- 「後でまとめて更新」と先送りする

**このルールの違反は重大な規約違反である**

## 【厳命】スクリプト・テンポラルファイルの取り扱い

### Pythonスクリプトの使用禁止

**Pythonスクリプトの作成・実行は原則禁止**

- ユーザーから明示的に「Pythonで」「Pythonスクリプトで」と指示がない限り、Pythonファイルを作成してはならない
- 自動化処理が必要な場合は、bashコマンドまたはsedなどのUNIXコマンドを使用すること
- Pythonに限らず、勝手にスクリプト言語（Ruby、Perl等）のファイルを作成してはならない

### テンポラルファイルの即時削除

**一時的な処理ファイルは実行後すぐに削除すること**

- シェルスクリプト、変換スクリプトなど、一時的な処理のために作成したファイルは、実行完了後すぐに削除する
- プロジェクトのルートディレクトリや作業ディレクトリに処理ファイルを残してはならない
- 削除を忘れると、リポジトリに不要なファイルが残り、コミット対象になる可能性がある

**正しい手順:**
1. テンポラルファイルを作成（必要な場合のみ）
2. 実行
3. **即座に削除**

**違反例（絶対禁止）:**
- Pythonスクリプトを勝手に作成して実行
- テンポラルファイルを実行後に削除せず放置
- 「後で削除」として先送りする

**このルールの違反は重大な規約違反である**

## 開発規約（必須遵守）

以下の規約を**必ず遵守**してください。
詳細は `docs/CODING_STANDARDS.md` を参照（作業前に必ず読み込むこと）。

### 開発フロー（最優先）

**作業順序（厳守）:**
1. 実装
2. TypeScript/ESLintチェック
3. Chrome MCPで実動作確認
4. コミット

**テスト:**
- **カバレッジ100%厳守**: テストしていないコードは絶対にコミットしない
- **Chrome MCPで実動作確認必須**: 動作確認完了までコミット禁止
- **テスト不可時は停止**: テストできない状況では作業を中断し、ユーザーに報告

理由: どんなに確信があっても、実際の動作確認なしでは予期せぬバグが混入する。

### 主要規約の概要

- **命名規則**: ドキュメント(UPPER_SNAKE_CASE.md)、ディレクトリ(lowercase)、コンポーネント(PascalCase)、変数/関数(camelCase)、定数(UPPER_SNAKE_CASE)
- **相対パス禁止**: `./`, `../` でのimportは禁止、必ずパスエイリアス使用（サービス固有の詳細は `docs/services/` 参照）
- **型定義・定数の配置**: 各サービスドキュメント参照
- **コードスタイル**: セルフドキュメンティングなコード、JSDocのみ、絵文字禁止
- **セキュリティ**: 環境変数必須、秘密情報のハードコード禁止
