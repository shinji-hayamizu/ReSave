# 脅威モデル

> 関連ドキュメント: [ビジネス要件](../requirements/business-requirements.md) | [アーキテクチャ](../requirements/architecture.md) | [データ保護](./data-protection.md) | [監査ログ](./audit-log.md)
> 最終更新: 2026-01-02

## 1. 概要

本ドキュメントでは、ReSaveの脅威モデルをSTRIDE分析手法に基づいて定義する。OWASP Top 10（2021）も考慮し、各脅威に対する対策とリスク評価を行う。

### 1.1 対象システム

| 項目 | 内容 |
|------|------|
| システム名 | ReSave - 忘却曲線記憶カードアプリ |
| 認証方式 | Supabase Auth (JWT) |
| データ保護 | PostgreSQL RLS、HTTPS |
| 対象ユーザー | 学生、語学学習者、ビジネスパーソン |

### 1.2 資産の分類

| 資産 | 機密性 | 完全性 | 可用性 | 説明 |
|------|--------|--------|--------|------|
| ユーザー認証情報 | 高 | 高 | 高 | メールアドレス、パスワードハッシュ |
| 学習データ | 中 | 高 | 高 | カード、タグ、学習履歴 |
| 統計データ | 低 | 中 | 中 | ストリーク、習熟度 |
| システム設定 | 中 | 高 | 中 | 通知設定、プロフィール |

## 2. STRIDE分析

### 2.1 Spoofing（なりすまし）

#### 脅威シナリオ

| ID | シナリオ | 攻撃経路 | 影響 |
|----|---------|---------|------|
| S-001 | 他ユーザーのJWTを盗用してセッションを乗っ取る | XSS、ネットワーク傍受 | 高 |
| S-002 | パスワードブルートフォース攻撃 | ログインエンドポイント | 高 |
| S-003 | フィッシングによる認証情報の詐取 | 偽サイト、偽メール | 高 |
| S-004 | OAuthトークンの不正取得 | リダイレクトURI操作 | 中 |

#### 対策

| 脅威ID | 対策 | 実装方式 | 優先度 |
|--------|------|---------|--------|
| S-001 | JWTの短期間有効期限設定 | Supabase Auth設定（1時間） | 必須 |
| S-001 | HttpOnly Cookieでトークン保存 | Supabase SSR設定 | 必須 |
| S-001 | Secure属性の強制 | HTTPS必須環境 | 必須 |
| S-002 | レート制限の実装 | Supabase Rate Limiting | 必須 |
| S-002 | アカウントロックアウト | 5回失敗で30分ロック | 必須 |
| S-003 | メール認証の必須化 | Supabase Email Confirmation | 必須 |
| S-003 | パスワードリセット時の本人確認 | リセットリンクの短期間有効期限 | 必須 |
| S-004 | OAuthリダイレクトURIの厳格な検証 | Supabase OAuth設定 | 必須 |

### 2.2 Tampering（改ざん）

#### 脅威シナリオ

| ID | シナリオ | 攻撃経路 | 影響 |
|----|---------|---------|------|
| T-001 | 他ユーザーのカードデータを改ざん | API直接呼び出し | 高 |
| T-002 | 学習履歴の不正操作（ストリーク詐称） | API直接呼び出し | 中 |
| T-003 | リクエストパラメータの改ざん | MITM攻撃 | 高 |
| T-004 | クライアント側でのデータ改ざん | DevTools操作 | 中 |

#### 対策

| 脅威ID | 対策 | 実装方式 | 優先度 |
|--------|------|---------|--------|
| T-001 | Row Level Security (RLS) の適用 | PostgreSQLポリシー | 必須 |
| T-001 | すべてのCRUD操作でuser_id検証 | RLSポリシー必須化 | 必須 |
| T-002 | サーバーサイドでの統計計算 | Edge Functions | 必須 |
| T-003 | HTTPS通信の強制 | Vercel/Supabase設定 | 必須 |
| T-003 | HSTS（HTTP Strict Transport Security）の設定 | レスポンスヘッダー | 必須 |
| T-004 | サーバーサイドバリデーション | Zod + Server Actions | 必須 |

**RLSポリシー例:**

```sql
-- ユーザーは自分のカードのみ操作可能
CREATE POLICY "Users can CRUD own cards"
ON cards FOR ALL
USING (auth.uid() = user_id)
WITH CHECK (auth.uid() = user_id);

-- 学習履歴は挿入のみ許可（改ざん防止）
CREATE POLICY "Users can only insert own reviews"
ON reviews FOR INSERT
WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can read own reviews"
ON reviews FOR SELECT
USING (auth.uid() = user_id);

-- 削除・更新は不可（監査証跡保持）
```

### 2.3 Repudiation（否認）

#### 脅威シナリオ

| ID | シナリオ | 攻撃経路 | 影響 |
|----|---------|---------|------|
| R-001 | 不正操作の否認（「私はやっていない」） | ログ不足 | 中 |
| R-002 | アカウント共有による責任逃れ | 認証情報共有 | 中 |
| R-003 | 自動化ツールによる操作の否認 | Bot利用 | 低 |

#### 対策

| 脅威ID | 対策 | 実装方式 | 優先度 |
|--------|------|---------|--------|
| R-001 | 監査ログの記録 | [監査ログ設計](./audit-log.md)参照 | 必須 |
| R-001 | ログにIPアドレス・User-Agent記録 | Edge Functions | 必須 |
| R-002 | セッション管理の強化 | 同時ログイン制限（オプション） | 推奨 |
| R-002 | ログイン通知メールの送信 | Supabase Hooks + Resend | 推奨 |
| R-003 | User-Agentの記録・分析 | 監査ログ | 推奨 |

### 2.4 Information Disclosure（情報漏洩）

#### 脅威シナリオ

| ID | シナリオ | 攻撃経路 | 影響 |
|----|---------|---------|------|
| I-001 | SQLインジェクションによるDB漏洩 | 入力値操作 | 高 |
| I-002 | エラーメッセージからの情報漏洩 | スタックトレース表示 | 中 |
| I-003 | APIレスポンスでの過剰な情報露出 | 不適切なフィールド返却 | 中 |
| I-004 | ログファイルからの機密情報漏洩 | ログへの秘密情報出力 | 高 |
| I-005 | 他ユーザーの学習データへの不正アクセス | IDOR (Insecure Direct Object Reference) | 高 |

#### 対策

| 脅威ID | 対策 | 実装方式 | 優先度 |
|--------|------|---------|--------|
| I-001 | パラメータ化クエリの使用 | Supabase Client（自動） | 必須 |
| I-001 | ORMレベルでのエスケープ | Supabase JS SDK | 必須 |
| I-002 | 本番環境でのエラー詳細非表示 | Next.js本番モード | 必須 |
| I-002 | カスタムエラーページの実装 | error.tsx | 必須 |
| I-003 | DTOによるレスポンス制御 | TypeScript型定義 | 必須 |
| I-003 | GraphQL/APIフィールド制限 | Supabase Select指定 | 必須 |
| I-004 | 機密情報のログ出力禁止 | ロガー設定、コードレビュー | 必須 |
| I-005 | RLSによるアクセス制御 | PostgreSQLポリシー | 必須 |
| I-005 | UUIDの使用（推測困難なID） | PostgreSQL gen_random_uuid() | 必須 |

### 2.5 Denial of Service（サービス拒否）

#### 脅威シナリオ

| ID | シナリオ | 攻撃経路 | 影響 |
|----|---------|---------|------|
| D-001 | 大量リクエストによるサーバー過負荷 | API連続呼び出し | 高 |
| D-002 | 大量のカード作成によるストレージ枯渇 | API経由での大量データ投入 | 中 |
| D-003 | 重いクエリによるDB過負荷 | 複雑な検索条件 | 中 |
| D-004 | 画像アップロードによるストレージ枯渇 | 大量・大容量ファイル | 中 |

#### 対策

| 脅威ID | 対策 | 実装方式 | 優先度 |
|--------|------|---------|--------|
| D-001 | レート制限の実装 | Supabase Rate Limiting | 必須 |
| D-001 | Vercel Edge Middlewareでの制限 | middleware.ts | 推奨 |
| D-002 | ユーザーごとのカード数上限設定 | RLSポリシー + トリガー | 必須 |
| D-002 | フリープランの制限（例: 1000カード） | DB制約 | 必須 |
| D-003 | クエリタイムアウトの設定 | PostgreSQL statement_timeout | 必須 |
| D-003 | インデックスの適切な設計 | DBマイグレーション | 必須 |
| D-004 | ファイルサイズ制限（5MB） | Supabase Storage設定 | 必須 |
| D-004 | ユーザーごとのストレージ上限 | Storage Policies | 必須 |

**レート制限設定例:**

| エンドポイント | 制限 | 理由 |
|---------------|------|------|
| 認証API | 10回/分 | ブルートフォース防止 |
| カード作成 | 100回/時 | スパム防止 |
| 画像アップロード | 20回/時 | ストレージ保護 |
| 一般API | 1000回/時 | 全般的な保護 |

### 2.6 Elevation of Privilege（権限昇格）

#### 脅威シナリオ

| ID | シナリオ | 攻撃経路 | 影響 |
|----|---------|---------|------|
| E-001 | 一般ユーザーが管理者権限を取得 | JWT操作、API脆弱性 | 高 |
| E-002 | RLSポリシーのバイパス | ポリシー設定ミス | 高 |
| E-003 | Edge Functionsの不正実行 | 認可チェック漏れ | 高 |

#### 対策

| 脅威ID | 対策 | 実装方式 | 優先度 |
|--------|------|---------|--------|
| E-001 | JWTクレームの検証 | Supabase Auth（自動） | 必須 |
| E-001 | ロールベースアクセス制御（RBAC） | Supabase roles（将来対応） | 推奨 |
| E-002 | RLSポリシーの厳格なテスト | マイグレーションテスト | 必須 |
| E-002 | デフォルト拒否ポリシー | すべてのテーブルにRLS必須 | 必須 |
| E-003 | Edge Functionsでの認証チェック | auth.uid()の必須検証 | 必須 |

**RLS有効化ガイドライン:**

```sql
-- すべてのテーブルでRLS有効化必須
ALTER TABLE cards ENABLE ROW LEVEL SECURITY;
ALTER TABLE tags ENABLE ROW LEVEL SECURITY;
ALTER TABLE reviews ENABLE ROW LEVEL SECURITY;
ALTER TABLE user_settings ENABLE ROW LEVEL SECURITY;

-- デフォルトはすべて拒否（ポリシー未定義=アクセス不可）
-- 必要な権限のみ明示的にポリシーで許可
```

## 3. OWASP Top 10（2021）対応

| 順位 | リスク | ReSaveでの対策 |
|------|--------|---------------|
| A01 | アクセス制御の不備 | RLS、認証必須化、RBAC準備 |
| A02 | 暗号化の失敗 | HTTPS必須、Supabase暗号化、bcrypt |
| A03 | インジェクション | パラメータ化クエリ、Zodバリデーション |
| A04 | 安全でない設計 | 脅威モデリング、セキュリティレビュー |
| A05 | セキュリティ設定ミス | デフォルト拒否、本番設定チェックリスト |
| A06 | 脆弱な古いコンポーネント | Dependabot、定期的な更新 |
| A07 | 認証の失敗 | Supabase Auth、MFA、レート制限 |
| A08 | ソフトウェアとデータの整合性 | CI/CDパイプライン保護、署名検証 |
| A09 | セキュリティログとモニタリングの失敗 | 監査ログ、Sentry、アラート設定 |
| A10 | サーバーサイドリクエストフォージェリ (SSRF) | 外部URLの検証、許可リスト |

## 4. リスク評価マトリクス

### 4.1 評価基準

**影響度:**

| レベル | 説明 |
|--------|------|
| 高 | データ漏洩、サービス停止、法的問題 |
| 中 | 一部機能停止、ユーザー体験低下 |
| 低 | 軽微な影響、迅速に復旧可能 |

**発生確率:**

| レベル | 説明 |
|--------|------|
| 高 | 攻撃が容易、頻繁に試行される |
| 中 | 専門知識が必要、ある程度の頻度 |
| 低 | 高度な技術が必要、稀にしか発生しない |

### 4.2 リスク一覧

| 脅威ID | 脅威 | 影響度 | 発生確率 | リスクレベル | 対策状況 |

| S-001 | JWTセッション乗っ取り | 高 | 中 | 高 | 対策済（設計） |
| S-002 | ブルートフォース | 高 | 高 | 高 | 対策済（設計） |
| T-001 | データ改ざん | 高 | 中 | 高 | 対策済（RLS） |
| I-001 | SQLインジェクション | 高 | 低 | 中 | 対策済（Supabase） |

## 5. セキュリティチェックリスト

### 5.1 開発時チェックリスト

- [ ] すべてのテーブルにRLSが有効化されているか
- [ ] RLSポリシーがデフォルト拒否になっているか
- [ ] ユーザー入力はすべてZodでバリデーションされているか
- [ ] 機密情報がログに出力されていないか
- [ ] エラーメッセージに内部情報が含まれていないか
- [ ] 環境変数で秘密情報が管理されているか

### 5.2 デプロイ時チェックリスト

- [ ] HTTPS が強制されているか
- [ ] HSTSヘッダーが設定されているか
- [ ] CORS設定が適切か（許可オリジン明示）
- [ ] レート制限が設定されているか
- [ ] エラー詳細が非表示になっているか
- [ ] Sentryが設定されているか

### 5.3 運用時チェックリスト

- [ ] 依存パッケージの脆弱性チェック（週次）
- [ ] 監査ログのレビュー（週次）
- [ ] 異常なアクセスパターンの確認（日次）
- [ ] バックアップの確認（日次）
